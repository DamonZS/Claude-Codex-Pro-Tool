# Harness Engineering 理论

## 1. 定义

Harness Engineering 是一种面向 AI 代理协作的软件开发工作法。

它的核心不是让 AI 自由发挥，而是为 AI 提供一套稳定的“工作夹具”：清晰上下文、明确规格、可验证验收标准、有限改动范围、真实验证证据和必要的角色分工。

在这套工作法里，AI 不是被要求“更像人一样猜需求”，而是被放进一个更清楚的任务系统中工作。好结果来自系统约束，而不是来自临场运气。

## 2. 要解决的问题

AI 编码失败通常不是因为不会写代码，而是因为任务系统松散：

- 目标不清，AI 会补脑。
- 范围不清，AI 会扩大改动。
- 验收不清，AI 会用“看起来可以”代替完成。
- 上下文不清，AI 会重复已有逻辑或破坏隐藏约束。
- 验证不足，AI 会把未运行的假设当成事实。
- 同一代理同时实现和裁判，容易产生自我确认偏差。

Harness Engineering 的目标，是把这些不确定性前置处理，让 AI 的输出更稳定、更可审计、更容易复现。

## 3. 核心目标

这套工作法追求五个目标：

- 可理解：代理先知道项目是什么、任务是什么、不能动什么。
- 可执行：规格足够明确，能直接指导实现。
- 可验收：完成标准在开发前已经定义。
- 可验证：结果由测试、构建、运行、截图、日志或命令输出证明。
- 可追溯：最终汇报能从需求一路追到代码、验证和剩余风险。

## 4. 核心原则

### 先理解，后修改

开发前必须先读项目规则、README、相关规格、验收标准和代码。理解不足时，优先补上下文，而不是马上写代码。

### 先规格，后开发

重要任务必须先有规格文档。规格文档回答“要做什么、不做什么、如何使用、有什么约束”。

没有规格时直接写代码，本质上是在让 AI 自己定义目标。

### 先验收，后交付

验收标准必须在交付前存在，最好在实现前存在。验收标准回答“怎样才算完成、需要什么证据”。

没有验收标准时，代理既当运动员又当裁判，质量会不稳定。

### 先验证，后汇报

不能在未验证时宣称完成。验证可以是测试、构建、本地运行、手动检查、截图、日志或命令输出，但必须真实发生。

### 最小必要改动

每次任务只改满足规格所需的内容。无关重构、顺手优化、扩大范围都会增加回归风险，也会让验收变得模糊。

### 证据优先于自信

代理的语气不等于事实。判断完成依赖证据，不依赖“应该可以”“看起来没问题”。

## 5. 三类核心文档

### `AGENTS.md`

项目地图和代理契约。它说明项目目的、目录结构、任务流程、验证命令、安全边界和交付格式。

代理进入项目后首先阅读它。

### `spec/*.md`

目标定义文档。它说明背景、目标、非目标、用户视角、功能要求、界面要求、数据接口、技术约束和交付范围。

规格文档让“想要什么”变成“可实现什么”。

### `acceptance/*.md`

完成判定文档。它说明每一项通过标准、验证方式和证据要求。

验收标准让“做完了”变成可检查的事实。

## 6. 任务分层

不是所有任务都需要同样重的流程。流程深度应该跟风险和不确定性匹配。

### 微任务

例子：改一个错别字、解释一段代码、运行一个查询命令。

要求：

- 可以不新增 spec。
- 可以用现有 `AGENTS.md` 和用户指令作为轻量规格。
- 仍然不能编造验证结果。

### 普通任务

例子：小型 UI 文案调整、局部 bug 修复、单个函数行为修正。

要求：

- 如果已有相关 spec 和 acceptance，必须阅读。
- 如果没有，可以在对话中形成轻量规格和验收口径。
- 至少运行一项定向验证。

### 重要任务

例子：新增页面、接口、模块、配置写入、数据处理逻辑、跨文件重构。

要求：

- 必须有 `spec/*.md`。
- 必须有对应 `acceptance/*.md`。
- 实现前总结上下文和风险。
- 验证必须覆盖主要验收项。

### 高风险任务

例子：安装器、更新流程、鉴权、密钥处理、数据迁移、删除操作、生产配置、第三方脚本执行。

要求：

- 必须有完整 spec 和 acceptance。
- 必须明确回滚或恢复策略。
- 必须使用更强验证，必要时加入评审代理。
- 未验证的风险必须在最终汇报中清楚标出。

## 7. 角色边界

复杂任务建议拆分角色，核心目的是分离责任。

### 规格代理

负责把模糊需求变成清晰规格。

负责：

- 梳理背景和目标。
- 明确范围与非范围。
- 补齐用户视角和功能要求。
- 写入或更新 `spec/*.md`。

不负责：

- 直接实现代码。
- 宣称任务完成。

### 验收代理

负责把规格变成可判定标准。

负责：

- 写入或更新 `acceptance/*.md`。
- 为每个验收项定义通过标准。
- 指定验证方式和证据。

不负责：

- 降低验收标准来适配实现结果。
- 用主观判断替代证据。

### 实现代理

负责按规格做最小必要改动。

负责：

- 阅读相关代码和测试。
- 实现规格内的内容。
- 添加必要测试或文档。
- 不改无关内容。

不负责：

- 擅自扩大功能范围。
- 绕过验收标准。

### 测试代理

负责验证实现是否满足验收标准。

负责：

- 设计或选择验证方式。
- 运行测试、构建或手动检查。
- 报告证据和失败项。

不负责：

- 把未运行的验证说成已通过。
- 忽略失败结果。

### 评审代理

负责审查风险和偏差。

负责：

- 检查是否偏离规格。
- 检查是否遗漏验收项。
- 检查回归、可维护性和安全风险。

不负责：

- 为了交付速度跳过问题。
- 替代真实测试。

## 8. 标准闭环

Harness Engineering 的完整闭环如下：

1. 需求进入：用户提出目标或问题。
2. 上下文读取：代理阅读项目规则、已有文档、代码和测试。
3. 规格化：把目标整理成 spec，明确范围和非范围。
4. 验收化：把 spec 转换成 acceptance，明确通过标准和证据。
5. 实施：按 spec 做最小必要改动。
6. 验证：按 acceptance 运行真实验证。
7. 交付：汇报修改、验证、验收映射和风险。
8. 复盘：必要时把新发现的约束补回文档。

这个闭环的关键，是每一步都产生下一步可使用的输入。

## 9. 适用场景

适合使用完整流程的场景：

- 多文件改动。
- 用户可见功能。
- 数据或配置写入。
- 安装、更新、发布、迁移。
- 安全敏感逻辑。
- 多代理协作。
- 用户要求可审计交付。

适合使用轻量流程的场景：

- 单次命令查询。
- 简短解释。
- 非代码文案微调。
- 明确、低风险、可立即验证的小改动。

不适合过度使用完整流程的场景：

- 用户只需要一个快速答案。
- 任务没有代码或文档落地。
- 验收标准显而易见且风险极低。

轻量流程不是跳过规则，而是把规则压缩到合适尺寸。

## 10. 常见失败模式与纠偏

### 失败模式：没有 spec 就开始写代码

纠偏：停止实现，先写或补齐规格文档。若任务很小，至少在对话中明确目标、范围和验收口径。

### 失败模式：spec 写得像愿望清单

纠偏：把抽象描述改成可执行要求。每条要求应能回答“代码要做什么”。

### 失败模式：验收标准不可验证

纠偏：为每个验收项补充通过标准、验证方式和证据。无法验证的要求不能作为完成标准。

### 失败模式：实现时顺手重构

纠偏：回到交付范围，只保留满足规格所需的改动。把额外想法记录为后续建议。

### 失败模式：测试失败后继续宣称完成

纠偏：明确报告失败命令、失败原因和剩余风险。能修则修，不能修则说明阻塞点。

### 失败模式：最终汇报只说“已完成”

纠偏：按交付格式说明修改内容、验证结果、验收映射和风险。

### 失败模式：子代理重复工作或互相覆盖

纠偏：给每个子代理明确输入、输出和禁止范围。子代理之间通过文档产物交接，而不是靠口头猜测。

## 11. 度量指标

可以用以下指标判断 Harness Engineering 是否有效：

- 需求返工率：交付后因理解偏差导致的返工是否减少。
- 验收命中率：最终结果满足 acceptance 的比例。
- 验证覆盖率：每次交付是否都有真实验证证据。
- 范围漂移率：任务中出现无关改动的频率。
- 缺陷逃逸率：已交付功能后续发现的问题数量。
- 文档复用率：后续任务是否能复用既有 spec、acceptance 和理论文档。
- 决策可追溯性：能否从最终改动追溯到规格和验收项。

这些指标不要求一开始就自动化，但应在复盘时被观察。

## 12. 方法论边界

Harness Engineering 不能保证 AI 永远正确。它降低的是不确定性，不是消除所有错误。

它不能替代：

- 工程师判断。
- 真实测试。
- 安全审查。
- 用户对目标的最终确认。
- 对生产环境的谨慎操作。

它能提供的是一套更可靠的协作结构，让错误更早暴露，让结果更容易检查。

## 13. 在本项目中的默认解释

在 Claude Codex Pro Tool 中，这套理论落地为以下默认规则：

- `AGENTS.md` 是所有代理的入口。
- 重要任务必须先有 `spec/*.md` 和 `acceptance/*.md`。
- 代码实现必须保持最小必要改动。
- 涉及安装器、配置、密钥、数据、更新和第三方脚本的任务按高风险处理。
- 交付必须包含验证证据和验收映射。
- 原有无关工作区改动不得被回滚。

当用户没有特别说明时，代理应按这些默认规则执行。用户明确要求快速回答或仅做查询时，可以采用轻量流程，但仍不得编造结果。

## 14. Skill 化与跨项目复用

当这套方法论被封装成 Codex Skill 后，它可以脱离单一仓库复用。默认 Skill 名称为 `$harness-engineering`。

适合显式调用 Skill 的场景：

- 用户希望新项目也采用先规格、后验收、再实现的流程。
- 任务涉及页面、接口、模块、重构、数据、配置、安装器、发布或安全敏感逻辑。
- 需要把模糊需求整理成 `spec/` 和 `acceptance/`。
- 需要设计子代理分工。
- 需要评审一个任务是否真正满足验收标准。

在跨项目中使用时，应遵循三层规则：

1. 如果项目有自己的 `AGENTS.md`、`CLAUDE.md` 或同类规则，优先遵守项目规则。
2. 如果项目没有规则，用 `$harness-engineering` 提供默认工作流。
3. 如果用户明确要求快速回答或一次性查询，可以采用轻量流程，但仍不得编造验证结果。

Skill 包应保持精简：

- `SKILL.md` 只放触发条件、核心流程、决策规则和必要提醒。
- `references/theory.md` 放理论细节。
- `references/templates.md` 放规格、验收、上下文总结和最终汇报模板。
- `references/checklists.md` 放执行检查清单。

维护 Skill 时，优先补充能改变代理行为的内容，避免把一次性项目记录写进 Skill。项目专属规则仍应留在项目自己的 `AGENTS.md`、`spec/` 和 `acceptance/` 中。
