# Harness Engineering 工作法文档基线

## 背景

本仓库需要一份项目级 AI 代理工作契约。如果缺少统一的代理指南、任务规格和验收标准，代理可能会不一致地推断需求、修改无关文件，或在没有验证的情况下宣称完成。

用户提供了一套 Harness Engineering 标准，要求代理先阅读项目上下文，再补齐规格文档，交付前定义验收标准，验证真实结果，并在复杂任务中使用子代理分工。

## 目标

本次要完成：

- 新增根目录 `AGENTS.md`，说明项目目的、目录结构、开发约束、任务流程、验证命令、子代理分工、安全边界和交付格式。
- 新增 `spec/` 目录，用于存放任务和功能规格文档。
- 新增 `acceptance/` 目录，用于存放匹配的验收标准。
- 将本文作为初始示例规格文档和本次文档基线任务的规格说明。
- 新增与本文匹配的验收标准。

本次不包含：

- 修改应用源码。
- 修改构建、发布、CI 或打包行为。
- 新增依赖。
- 重写现有 README 或 docs 内容。
- 为登录、订单查询、后台仪表盘等未来无关功能创建示例规格，除非用户另行提出。

## 用户视角描述

AI 编码代理进入仓库后，可以打开 `AGENTS.md` 并立即理解：

- 项目是什么。
- 源码、文档、规格文档、验收标准、测试和脚本分别在哪里。
- 编码前必须阅读什么。
- 规格文档和验收标准应如何组织。
- 可以使用哪些验证命令。
- 用户期望最终交付说明包含哪些内容。

维护者可以在未来把任务规格文档放到 `spec/` 下，并把匹配的验收标准放到 `acceptance/` 下，采用一一对应的命名方式。

## 功能要求

- `AGENTS.md` 必须存在于仓库根目录。
- `AGENTS.md` 必须用 1 到 3 段实用文字描述项目。
- `AGENTS.md` 必须说明本仓库关键目录。
- `AGENTS.md` 必须要求代理在开发前阅读 `AGENTS.md`、`README.md`、相关规格文档、匹配的验收标准和相关代码。
- `AGENTS.md` 必须说明缺少规格文档或验收标准时，应先补齐再做重要实现工作。
- `AGENTS.md` 必须说明合格规格文档应包含哪些内容。
- `AGENTS.md` 必须说明 `spec/` 与 `acceptance/` 的一一对应命名关系。
- `AGENTS.md` 必须说明从理解任务到交付的标准流程。
- `AGENTS.md` 必须包含推荐的 Super Powers 子代理角色及职责。
- `AGENTS.md` 必须包含与当前仓库匹配的本地安装、开发、构建和验证命令。
- `AGENTS.md` 必须包含与本项目相关的安全边界。
- `spec/harness-engineering-docs.md` 必须定义本次文档基线的范围。
- `acceptance/harness-engineering-docs.md` 必须定义本次文档基线的可验证完成标准。

## 界面 / 交互要求

本次不包含应用 UI 改动。

文档应在 Markdown 中易读，标题清晰，示例可直接复制。

## 数据与接口要求

本次不修改运行时数据结构、数据库 schema 或 API。

## 技术约束

- 本任务不得编辑应用源码。
- 不修改现有 README 内容。
- 不引入依赖或生成产物。
- 文档应保持实用，并贴合当前仓库。
- 使用能在 GitHub 和本地编辑器中正常渲染的普通 Markdown。

## 交付范围

- `AGENTS.md`
- `spec/harness-engineering-docs.md`
- `acceptance/harness-engineering-docs.md`
- 验证这些文件存在，并包含所需章节。
