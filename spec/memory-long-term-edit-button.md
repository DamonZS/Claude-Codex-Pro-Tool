# 盘古记忆经验教训查看与编辑入口

## 背景

盘古记忆总览之前以“长期记忆”和条数展示为主，容易让用户只看到一个总数，而看不到每一条真正沉淀出来的经验教训。后端已经具备 `update_memory_assist_item` 能力，需要把保存项在前端重新呈现为“经验教训”，并保持现有记忆库、注入链路和安全边界不变。

本次用户明确要求不要生成很多条散乱长期记忆，而是把当前已经保存的内容合成一条精简经验教训手册。压缩操作只允许作用于 `memory_items` 保存项，执行前必须备份 `memory_assist.sqlite`，不得删除数据库文件、采集日志、候选记录或会话证据。

## 目标

本次要完成：

- 在盘古记忆总览中把保存项命名为“经验教训”，并提供“查看/编辑”按钮。
- 点击按钮后在概览页右侧详情区域展示一条精简经验教训手册，并刷新记忆数据。
- 提炼结果必须合成为一条手册型经验教训，不得把每个片段分别写成很多条 `memory_items`。
- 前端不再在经验教训入口、详情标题或列表标题中展示条数，只保留总入口和列表内容。
- 盘古记忆总览中的经验教训、采集记录、工作区、数据库、最近备份必须使用紧凑矩阵布局展示，避免纵向堆叠过高。
- “待确认”不再作为盘古记忆主界面的可见状态或列表展示；低置信候选机制可保留在后端作为安全缓冲。
- 在盘古记忆区域提供“提炼经验教训”按钮，显式触发历史会话高置信内容提炼。
- 点击“提炼经验教训”后必须立即显示运行反馈，说明正在使用 Codex 本地 SQLite、rollout 会话文件和 `memory_assist.sqlite`。
- 提炼链路必须遍历可见 Codex 工作区与会话历史，而不是只做少量最近记录抽样。
- 提炼完成后必须输出结果摘要，包括检查到的 DB、会话文件、用户消息、采集记录、自动写入经验教训、候选和错误数量。
- 经验教训手册可进入编辑状态，修改文本与分类后保存。
- 保存时保留原有 workspace、tags、source、sourceSessionId 等元数据。
- 按用户要求备份数据库后把当前 `memory_items` 压缩为一条 `lesson-manual` 手册。

本次不包含：

- 不修改 Claude 中文注入。
- 不删除、重置或替换 `memory_assist.sqlite` 文件。
- 不清空采集日志、候选记录、自检日志或会话证据。
- 不改变经验教训自动学习、候选确认、采集日志、自检算法。
- 不做无关 UI 重构。

## 用户视角描述

用户在概览页看到盘古记忆的“经验教训”入口后，可以点击“查看/编辑”，在右侧详情区域直接展开一条精简经验教训手册。用户可以编辑这本手册的内容或分类，保存后列表刷新并展示最新内容。用户取消编辑时，不应写入任何数据。

## 功能要求

- 概览页经验教训区域必须提供明确按钮，文案为“查看/编辑”。
- 按钮行为必须在概览页右侧详情区域展开经验教训列表，并触发盘古记忆刷新。
- 盘古记忆总览、右侧经验教训详情和盘古记忆管理面板应提供“提炼经验教训”按钮。
- “提炼经验教训”必须调用真实后端提炼链路，从真实 Codex 历史会话与采集证据中提取高置信经验教训，并刷新列表。
- “提炼经验教训”开始时必须展示“正在使用 Codex 本地 SQLite、rollout 会话文件、memory_assist.sqlite 遍历工作区与会话”的运行态提示。
- “提炼经验教训”结束时必须展示历史扫描结果摘要，让用户知道实际检查了多少 DB、会话文件、用户消息、采集和经验教训写入。
- “提炼经验教训”必须写入 `manager.memory.selfcheck.start`、`manager.memory.selfcheck.result` 或 `manager.memory.selfcheck.failed` 诊断日志。
- 前端按钮点击必须写入 `manager.ui.button.click`，通用前端动作包装器必须写入 `manager.ui.action.start`、`manager.ui.action.result` 或 `manager.ui.action.failed`，用于覆盖管理工具按钮点击后的日志证据。
- 概览页右侧详情区域只展示合成后的经验教训手册，不再展示多条片段。
- 经验教训入口、详情标题和列表标题不得展示“X 条”条数文案；条数仅允许保留在内部日志或后端统计中。
- 盘古记忆总览信息区必须使用矩阵布局；操作按钮应紧凑排列，不再与信息项混在同一个纵向列表中。
- 盘古记忆前端不得展示“待确认”行、待确认列表或确认/忽略候选按钮。
- 编辑状态必须包含手册文本编辑框、分类输入框、保存按钮和取消按钮。
- 保存按钮在文本为空时不可用。
- 保存成功后必须刷新盘古记忆状态与列表。
- 删除经验教训的既有能力必须保留。
- 按用户本次要求，交付前必须备份 `memory_assist.sqlite` 并把当前 `memory_items` 压缩为一条 `lesson-manual`，压缩后验证 `memory_items` 为 1。

## UI / 交互要求

- 入口按钮使用现有按钮样式与图标体系，不新增 UI 库。
- 编辑控件应在单条经验教训卡片内展开，避免跳转或弹出额外窗口。
- 详情区域应位于概览页右侧诊断与修复卡片下方，利用该列空白区域展示多条内容。
- 文本区域应使用现有 `ops-textarea` 样式，保持与当前管理工具视觉一致。
- 手册为空时显示“暂无经验教训。”。
- 经验教训手册必须使用独立边框和背景承载，内容是精简条目化手册，而不是大量散乱卡片。
- 概览页红框信息区应在桌面宽度下两列展示信息项，操作按钮三列展示；窄屏时允许自动回落为单列。

## 数据与接口要求

- 前端调用现有 Tauri 命令 `update_memory_assist_item`。
- 请求结构为 `{ id, item }`，其中 `item` 至少包含 `text`、`workspace`、`category`、`tags`、`source`、`sourceSessionId`。
- 保存失败时沿用现有通知机制展示后端返回消息。
- 提炼经验教训复用现有 `run_memory_assist_selfcheck` 的 `repair: true` 路径，保证进入同一套历史采集、过滤、脱敏、SimHash 合并和摘要缓存刷新链路。
- `run_memory_assist_selfcheck` 的提炼路径必须记录后端诊断事件，日志 detail 中包含 `repair`、最终状态、是否修复、检查项和历史扫描摘要。
- 预览模式也应模拟该命令，保证前端预览不报错。
- 提炼完成后的压缩操作允许先读取全部 `memory_items`，再用单条 `lesson-manual` 记录替换它们；不得删除数据库文件或其他表。

## 技术约束

- 只修改盘古记忆前端管理链路、必要测试、规格与验收文档。
- 不新增依赖。
- 不改变后端数据库 schema。
- 不删除现有功能入口。
- 不修改 Claude 中文注入和 Codex 注入脚本。

## 交付范围

- 本规格文档。
- 对应验收标准。
- 管理工具前端入口与编辑 UI 文案。
- 前端预览 bridge 的命令模拟若受影响则同步更新。
- 数据库备份与当前 `memory_items` 压缩操作记录。
- 定向测试与构建验证。
