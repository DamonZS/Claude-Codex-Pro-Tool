# 管理工具启动与导航响应性修复

## 背景

管理工具打开后会并发加载概览、公告、Claude 状态、Claude 汉化窗口状态、Claude Desktop 开发模式、设置和插件仓库状态。当前仍有启动命令以同步 Tauri command 形式执行系统探测或磁盘读取；一旦该调用耗时，Windows 窗口消息泵会被阻塞，标题出现“未响应”，页面点击全部失效。

## 目标

- 管理工具启动和切回概览时保持窗口可交互。
- 所有可能执行系统探测、目录扫描或文件读取的启动命令必须在 Tauri blocking 线程池中执行。
- 保留现有概览、设置、Claude 汉化状态和通知内容。
- 后端失败仍返回明确状态，不伪装成功。

## 非目标

- 不修改 Codex/Claude 进程、注入脚本或盘古记忆数据库。
- 不取消启动时必要的数据刷新。
- 不重构整个前端加载框架。

## 功能要求

- `load_settings` 改为异步命令，并把设置读取与用户脚本库存读取整体放入 `spawn_blocking`。
- `load_claude_chinese_window_status` 改为异步命令，并把 `detect_status_light` 等同步系统探测放入 `spawn_blocking`。
- blocking 任务 join 失败时必须返回结构完整的失败 payload。
- 不在 UI 主线程等待外部网络、进程探测、递归目录读取或大文件读取。
- 保留现有前端并发加载行为；后端命令应允许结果独立完成并回传。

## UI / 交互要求

- 启动后窗口不能出现持续“未响应”。
- 用户在加载期间可点击导航；已有加载态和通知逻辑保持可用。
- 加载失败时显示原有模块错误信息，不把整个界面冻结。

## 技术约束

- 不新增依赖。
- 只结束并重启管理器进程，不结束 Codex 或 Claude。
- 添加回归测试，防止启动命令再次退化为同步阻塞实现。

## 交付范围

- Manager Tauri 命令异步化。
- Windows UI/命令源码回归测试。
- 本规格与匹配验收文档。
