# 关于页更新检查与下载体验

## 背景

管理工具“关于”页当前只通过 GitHub Release 的 `latest.json` 下载资源检查版本。当 Release 资源域连接缓慢或不可达时，“检查更新”会长时间等待并最终报错。安装包下载又会整包读入内存，前端在完成前收不到状态或进度，用户无法判断按钮是否生效。

## 目标

- 让检查更新并发使用 GitHub Releases API、`releases/latest` Tag 重定向和 `latest.json`；任一可信来源成功即可返回。
- 系统代理缓慢或失效时，同时尝试直连可信 GitHub 地址，避免被单一网络路径拖死。
- 首轮网络路径均失败时，仅对轻量 Tag 重定向执行一次有限重试；成功结果短时缓存，避免重复点击持续请求 GitHub。
- 点击检查或下载后立即显示操作状态，防止重复提交。
- 安装包采用分块下载并向前端报告阶段、字节数和百分比。
- 下载未完成或失败时不覆盖已有完整安装包。
- 保持关于页现有布局和安装包来源校验，不改动其他页面。

## 用户视角

- 点击“检查更新”后，按钮立即进入“检查中”，不再像未响应。
- GitHub Release 资源域异常时，只要 GitHub API 可访问，仍能快速取得最新版本信息。
- 点击“下载并运行安装包”后，可看到连接、下载、启动安装包三个阶段以及实时进度。
- 下载期间两个更新按钮不可重复点击；失败后恢复可操作状态并显示明确错误。
- Tag 回退没有发布正文时，页面仍明确显示已取得版本和安装资源，不显示“未检查”。

## 功能要求

1. 最新版本检查同时具备 GitHub Releases API、`releases/latest` Tag 重定向和 `latest.json` 三个来源，任一可信来源成功即可返回。
2. 当 API 匿名配额耗尽且 Release asset 下载域不可用时，Tag 重定向仍可在不下载资源正文的情况下返回版本，并按固定仓库、Tag 与当前平台规则生成安装包地址。
3. 检查请求使用独立的短连接和总请求超时，不沿用普通业务请求的长等待时间。
4. 检查与安装包连接同时竞速系统代理和直连路径；任一路径成功即可继续，不能修改或清理用户代理设置。
5. 首轮检查全部失败时，仅自动重试系统代理和直连的 Tag 重定向；成功结果在当前进程短时缓存。
6. 下载使用流式分块写入同目录 `.part` 文件，不一次性缓存完整安装包。
7. 下载完成并刷新文件后再替换正式安装包；下载失败时清理 `.part`，保留已有正式文件。
8. 下载进度至少包含 `connecting`、`downloading`、`launching`、`complete`、`failed` 阶段，以及已下载字节、总字节和可计算时的百分比。
9. Tauri 通过 `update-download-progress` 事件将进度发送给当前 Manager 窗口。
10. 前端调用失败时必须退出运行态，不能永久显示下载中。

## UI / 交互要求

- 不改变关于页双栏和面板结构。
- 检查按钮在请求期间显示旋转图标和“检查中”。
- 下载按钮在运行期间显示当前阶段和百分比。
- 更新面板内增加紧凑进度条、已下载大小和总大小；总大小未知时显示不确定进度。
- 两个按钮具有明确按压缩放反馈，禁用时不触发反馈。
- 进度区域使用 `aria-live` 和 `progressbar` 语义。

## 数据与接口

- Core 新增可信 GitHub API 地址常量和带回退的 Release 获取函数。
- Core 新增可序列化的更新进度结构，以及带进度回调的安装函数；保留原 `perform_update` 兼容入口。
- `perform_update` Tauri 命令增加 `AppHandle` 注入，但前端参数仍只提交 `expectedVersion`。
- 前端 `UpdateResult` 增加可选的阶段、已下载字节和总字节字段。

## 技术约束

- 不新增第三方依赖，复用 reqwest、futures-util、Tauri event 和现有 Button。
- Release 安装包仍必须来自固定 GitHub 仓库、HTTPS 地址并匹配当前平台。
- 不修改发布流程、版本规则、安装器启动方式和其他页面 UI。
- 不回滚工作区中已有的未提交修改。

## 交付范围

- Core Release 查询、流式下载与进度回调。
- Tauri 进度事件。
- 关于页即时反馈、按钮状态和进度显示。
- 定向测试、前端检查、默认 `target/release` 全量构建和本地启动。
