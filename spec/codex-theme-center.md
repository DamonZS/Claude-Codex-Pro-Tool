# Codex 主题中心

## 背景

当前管理工具已经具备 Codex 启动、注入、翻译补丁、模型注入、供应商切换、会话与维护等能力，但缺少一个面向“主题资产”的独立产品入口。现有 UI 外观主题仅覆盖管理工具壳层的浅色 / 深色 / 跟随系统，不等同于用户希望管理的 Codex 界面主题。项目需要增加一个只负责 Codex 主题导入、展示、应用、恢复与回滚的主题中心，同时保持既有业务、Tauri IPC、注入链路与非主题行为稳定。

本次规格定义一个独立一级导航的“主题中心”，让用户可以看到自己已拥有的全部 Codex 主题、导入新的主题资源、将主题应用到本地 Codex，并在需要时可靠恢复到 Codex 默认主题原貌。

## 目标

### 本次必须完成

- 新增独立一级导航“主题中心”，不与供应商、会话、插件、设置或现有管理工具壳层主题菜单混用。
- 页面顶部提供“新建主题”主操作，用于导入本地主题包。
- 页面默认展示“全部已拥有主题”，不区分官方 / 导入 / 当前应用作为默认首屏筛选条件。
- 桌面端以“每行三张、稳定尺寸、图片主导”的悬浮卡片网格展示主题；窄屏时按响应式规则降级为双列或单列。
- 主题卡片使用的预览必须是该主题版本在同一 Codex 版本真实 Renderer 中运行后采集的截图；版本、运行时载荷哈希或采集凭据不匹配时展示明确的待采集 / 已失效空态，不得回退到设计稿、静态示意图或旧版本截图。
- 第一张卡片永久为“Codex 默认主题”，且该主题必须可真实恢复到用户本机 Codex 的默认原貌，而不是仅做视觉占位。
- 用户点击非默认主题应用成功后，界面明确提示“需重启 Codex 后生效”或等价文案；应用失败时不得显示成功提示。
- 只要当前主题不是 Codex 默认主题，启动器就必须主动启用完整前端注入链，不受其他增强功能开关是否关闭影响。
- 从 Manager 执行“启动 / 重启 Codex”后，当前主题必须随新文档脚本进入目标 Renderer，并留下可定位主题 ID、generation、端口和注入结果的诊断记录。
- 明确定义主题清单、manifest、预览图、样式资源、导入校验、临时目录、原子替换、回滚与错误反馈的数据结构和流程。
- manifest 必须支持受控的 `css_variables`、`root_attributes`、`asset_variables`，并由独立 Theme Loader 按声明精确应用；本地图片进入前端或 Renderer 前必须经校验并转换为 Data URI。
- 对包内路径、文件类型、魔数 / MIME、文件和解包总大小、文件数量、主题及资源命名执行后端强制校验，任何未知、歧义、越界或超限输入均不得进入正式主题库。
- 明确定义借鉴 Codex-X 的会话事务、Mutation Journal、占用检测与恢复测试机制，并将其用于主题资源写入、版本切换与回滚。
- 明确定义借鉴 Dream Skin macOS 的 CDP 身份校验、动态端口发现、Renderer 指纹确认与真实截图验收边界，用于验证主题确实作用于目标 Codex Renderer。
- 明确要求主题应用链路与既有汉化、模型、输入框、菜单、状态条等注入能力相互隔离，互不覆盖、互不篡改。
- Theme Loader 必须幂等地精确收敛到当前 manifest，能修复自身样式节点、变量、根属性和资源绑定的缺失、重复或篡改，并在切换 / 恢复时只撤销自身拥有的状态。
- 明确禁止修改用户输入内容、供应商配置、模型配置、Base URL、API Key、Prompt 及其他非主题设置。
- 明确 Provider 导入、配置预览、签名更新只作为既有能力边界存在，本主题任务不得改动凭据处理。
- 明确凭据必须继续保持遮罩显示、日志脱敏和系统存储策略不变。
- 保留现有业务流程、现有 Tauri IPC 合约和所有非主题行为。

### 本次不包含

- 不改造现有供应商、模型、会话、插件、维护、翻译补丁、模型注入或重启链路的业务语义。
- 不新增云端主题商店、远程下载主题、账号同步、订阅或在线鉴权。
- 不把管理工具自身浅色 / 深色外观菜单改造成 Codex 主题中心的一部分。
- 不修改用户已有 Prompt、输入框内容、供应商配置文件或 API 凭据存储格式。
- 不复用汉化补丁或模型标识注入使用的 DOM 扫描逻辑作为主题注入实现。
- 不重写现有 Tauri command 名称、参数、返回结构或前端全局状态中枢。

## 用户视角描述

### 浏览主题

- 用户从左侧一级导航进入“主题中心”后，默认看到自己当前已拥有的全部主题。
- 页面首屏第一张卡片固定为“Codex 默认主题”，其后按导入时间倒序或最近使用顺序展示其他已拥有主题，具体排序规则需在实现阶段固定且保持稳定。
- 用户无需先切换筛选器即可浏览所有主题。
- 卡片预览只能展示与该卡片主题版本、当前 Codex 版本及已编译运行时载荷一致的真实 Codex 截图；证据缺失、损坏或过期时，预览区保持稳定尺寸并显示“等待真实运行截图”或等价空态。

### 新建主题

- 用户点击页面顶部“新建主题”按钮后，选择本地主题包或本地主题目录导入。
- 导入前系统执行 manifest、真实截图元数据、样式资源、受控变量、本地图片和文件结构校验；校验失败时明确报错并保持现有主题列表不变。
- 导入成功后，新主题出现在主题列表中，并具备预览图、名称、作者、版本、来源、更新时间等基础信息。

### 应用与恢复

- 用户点击“Codex 默认主题”卡片时，可将当前主题恢复为默认原貌。
- 用户点击其他主题卡片的“应用主题”后，系统执行隔离导入、临时目录验证、原子替换、上一版本保留和必要备份。
- 应用成功后，界面必须提示需要重启 Codex；若当前主题已是目标主题，则界面给出“已在使用中”或等价状态，不重复执行破坏性写入。
- 如果主题应用中任一步骤失败，界面必须显示失败结果，并确保用户仍停留在先前有效主题或已自动回滚到可用状态。
- 应用后的 Theme Loader 必须把 manifest 声明精确编译并收敛到唯一的主题挂载状态；检测到自身状态损坏时自动修复，恢复默认时只清理它实际拥有的主题状态。

## 功能要求

### 1. 导航与页面定位

- “主题中心”必须是管理工具左侧一级导航中的独立入口。
- 它不得被放入设置二级页、工具页子分组或现有概览卡片跳转中替代一级入口。
- 保留现有全部一级导航和兼容路由能力；新增主题中心不得破坏既有路由解析与刷新逻辑。

### 2. 页面顶部操作

- 页面顶部固定提供一个主按钮：“新建主题”。
- 可选提供次级操作，例如“刷新列表”“打开主题目录”“查看当前生效主题”，但不得弱化“新建主题”的主操作地位。
- “新建主题”触发系统文件选择器或现有受支持导入流程，不得要求用户手工编辑配置文件。

### 3. 默认展示规则

- 进入页面后默认数据集为“全部已拥有主题”。
- “已拥有主题”定义为：默认主题 + 已成功导入并通过校验的本地主题。
- 默认筛选不得隐藏当前未应用的主题，也不得只显示“已启用”主题。
- 当主题库中除内置默认主题外没有任何用户主题时，页面仍显示第一张默认主题卡片，并在网格剩余区域显示独立空态及“新建主题”操作；不得渲染空白页面、伪造额外主题或把默认卡片本身替换为空态。

### 4. 卡片布局与视觉要求

- 桌面宽屏下每行固定展示三张卡片。
- 卡片尺寸必须稳定，不得因标题长短、描述长短或预览图尺寸差异导致瀑布流高低跳动。
- 卡片视觉以预览图为主导，文本信息作为次级补充；预览区应占卡片主要面积。
- 预览区只能消费主题中心登记的真实 Codex 截图记录。记录必须同时绑定主题 ID、主题版本、Codex 版本 / 构建指纹、运行时载荷 SHA-256、截图 SHA-256、采集时间和脱敏 Renderer 采集标识；任一绑定不一致即按空态处理。
- 截图必须来自通过动态端口、进程身份和 Renderer 指纹校验的真实 Codex 主 Renderer，不得使用主题作者设计稿、CSS 合成图、DOM 夹具、Mock 页面、静态 HTML 或其他应用截图。
- 卡片采用带轻微悬浮感的面板样式，但不得使用营销化 Hero、大幅渐变、轮播或不稳定动画。
- 窄屏响应规则：
  - 中等宽度降为两列；
  - 更窄宽度降为单列；
  - 任何断点下都必须保持卡片宽高比稳定、按钮可点击、预览图不被拉伸变形。

### 5. Codex 默认主题卡片

- 列表第一张卡片永久保留给“Codex 默认主题”。
- 该卡片不得被删除、覆盖、隐藏、排序到其他位置，或被导入主题替换。
- 该卡片代表“恢复到 Codex 默认原貌”的真实可执行能力，必须依赖首次基线采集、安装时快照或首次应用主题前的默认资产备份，以及 Loader 所有权记录，而不是仅将主题 ID 切换为 `default`；基线与快照只覆盖主题功能实际拥有或改动的状态，不得整包覆盖 Codex 或其他增强能力的文件 / DOM 状态。
- 如果默认资产基线缺失、损坏或不可恢复，界面必须显示“默认主题不可恢复”之类的失败状态，并阻止误报成功。
- 默认主题预览同样遵守同版本真实截图约束。当前 Codex 版本尚未采集默认外观、截图损坏或截图属于旧 Codex 构建时，默认卡片保留且显示稳定的“暂无同版本真实截图”空态，不得借用自定义主题截图或内置示意图。

### 6. 主题导入校验

导入至少需要校验以下项目：

- 存在且可解析的 manifest 文件。
- manifest 中的 `id`、`name`、`version`、`format_version`、`author`、`preview`、`assets`、`entry_style` 等必需字段完整；可选 `css_variables`、`root_attributes`、`asset_variables` 必须符合本规格的受控结构，未知顶层执行性字段直接拒绝。
- `id` 必须匹配 `^[a-z0-9](?:[a-z0-9_-]{0,62}[a-z0-9])?$`，`version` 必须是规范 SemVer；展示名称和作者去除首尾空白后为 1 至 80 个 Unicode 标量且不含控制字符、双向控制符或路径分隔符。
- 所有资源路径使用 `/` 分隔的 UTF-8 相对路径，每个路径段必须匹配 `^[A-Za-z0-9][A-Za-z0-9._-]{0,79}$`，并在规范化、Unicode NFC、Windows 不区分大小写比较后仍唯一；拒绝空段、`.`、`..`、反斜杠、绝对 / UNC / 盘符路径、ADS 冒号、NUL / 控制字符、尾随点或空格、Windows 保留设备名、符号链接、重解析点、硬链接和通配符。
- 资源路径深度最多 8 层，规范化相对路径最多 240 UTF-8 字节，每段最多 80 UTF-8 字节；每个资产必须由 `assets` 显式逐项声明，禁止 glob、隐式目录扫描和未声明伴随文件。
- 只允许 UTF-8 JSON (`.json`)、UTF-8 CSS (`.css`)、PNG (`.png`)、JPEG (`.jpg` / `.jpeg`)、WebP (`.webp`) 和 WOFF2 (`.woff2`)；扩展名、解析结果、文件头魔数和探测 MIME 必须一致。SVG、HTML、JavaScript、可执行文件、归档嵌套及未知类型一律拒绝。
- 预览截图必须是可解码 PNG，像素尺寸在 `960x600` 至 `3840x2160` 之间且不超过 12 百万像素；其声明尺寸、截图哈希、主题版本、Codex 版本 / 构建指纹和运行时载荷哈希必须与实际记录一致。
- 限额固定为：归档压缩体积不超过 32 MiB、解包总大小不超过 64 MiB、文件数不超过 256、manifest 不超过 256 KiB、单个 / 全部 CSS 不超过 2 MiB / 4 MiB、单张图片不超过 8 MiB 且不超过 40 百万解码像素、单个 WOFF2 不超过 5 MiB、全部原始图片不超过 32 MiB、生成后的 Data URI 运行时载荷不超过 48 MiB。边界按字节精确计算，不得依赖客户端上报值。
- 样式资源存在且可由 CSS 解析器完整解析；`url(...)` 只能引用已声明的本地图片 / WOFF2，拒绝 `http:`、`https:`、`file:`、`javascript:`、`blob:`、网络相对地址、作者直接提供的 `data:` 以及 `@import`。
- 归档导入时必须防止 zip slip、绝对路径写入、符号链接越界和保留文件名覆盖。
- 重复 `id` 导入时，系统必须要求覆盖确认、另存为新版本或显式失败；不得静默覆盖已安装主题。
- 覆盖导入当前正在生效的主题时，文件替换与活动主题 `generation` 递增必须在同一事务中提交；下一次 Renderer 注入必须读取新版本 manifest、CSS 与资源，不得继续沿用旧载荷。覆盖未生效主题不得改变当前主题或无故递增 `generation`。
- 任何校验失败都不得写入正式主题目录、不得更新主题清单、不得显示导入成功。
- 导入校验、安装写入和回滚恢复必须记录到可审计的 Mutation Journal，至少包含操作 ID、主题 ID、阶段、目标路径、快照信息、结果和时间戳。

### 7. 临时目录、原子替换与回滚

- 导入与应用必须先写入受控临时目录，再进入正式目录或正式应用路径。
- 临时目录必须位于应用自有状态目录下，例如 `~/.claude-codex-pro/tmp/themes/<operation-id>/` 或平台等效位置。
- 正式主题库存目录与运行时应用目录必须分离，避免“导入即生效”。
- 主题安装与更新必须遵循“临时目录验证 → 原子替换 → 保留上一版本”的固定流程；上一版本在新版本完成验收前不得被提前删除。
- 写入流程必须具备借鉴 Codex-X 的会话事务语义：一次主题导入 / 应用 / 恢复视为单个事务，事务未提交前不得更新当前生效主题指针。
- 写入流程必须具备借鉴 Codex-X 的占用检测：若目标资源、活跃目录或回滚快照被当前 Codex 进程、Manager 进程或其他文件句柄占用，系统必须中止提交并返回明确失败，而不是部分覆盖。
- 应用主题时至少遵循以下顺序：
  1. 校验目标主题资源完整；
  2. 记录事务开始与 Mutation Journal 初始条目；
  3. 采集当前可恢复快照与上一版本引用；
  4. 将目标资源写入临时应用目录；
  5. 完成临时目录内完整性验证；
  6. 执行原子替换或等价的一次性切换；
  7. 写入当前生效主题记录；
  8. 执行恢复测试与结果记录；
  9. 清理临时目录；
  10. 向前端返回真实结果。
- 如任一步骤失败，系统必须：
  - 保留原主题继续可用，或
  - 自动从本次操作前快照回滚到上一个稳定状态。
- 若回滚也失败，必须向前端返回“应用失败且回滚失败”的明确错误，不得显示成功。
- 回滚逻辑必须具备借鉴 Codex-X 的恢复测试：回滚完成后必须校验活跃主题指针、目标文件完整性和上一稳定版本可读取性，不能只以文件复制成功作为恢复完成依据。

### 8. 主题应用与既有注入能力隔离

- 主题中心只处理主题 manifest、预览资源、样式资源和当前主题状态。
- 它不得覆盖、删除、重写或拼接既有汉化补丁、模型注入、输入框增强、菜单增强、状态条注入、Provider Sync、会话管理或其他注入脚本逻辑。
- 若主题最终作用于注入资源或本地静态资源层，必须采用可组合、可隔离的挂载点，例如独立主题样式入口、独立资源目录、独立主题标记或受控拼接顺序。
- 主题应用流程不得改写现有翻译文本资源、模型白名单注入逻辑或现有 bridge / helper token 流程。
- Renderer 主题注入必须作为独立视觉层存在，禁止复用汉化补丁、模型标识、输入框增强、菜单增强或其他现有注入能力使用的 DOM 扫描器、节点匹配器、事件监听器、MutationObserver 或选择器巡检逻辑。
- 若需要识别目标 Renderer，必须通过独立主题层挂载点或显式主题容器完成，而不是沿用汉化、模型、输入框或菜单注入的扫描入口；主题脚本不得接管、克隆、移动或删除这些能力拥有的 DOM 节点。
- Theme Loader 以规范化 manifest 编译出的运行时载荷为唯一事实源，按 `entry_style`、`css_variables`、`root_attributes`、`asset_variables` 和 generation 精确应用；不得通过扫描 CSS 文本猜测根类，也不得把“样式节点存在”视为完整激活。
- Loader 每次应用都必须收敛到且仅保留一个 CCP 主题 `<style>` 节点，节点文本 / 哈希、受控 CSS 变量、根类、根数据属性、资产 Data URI、主题 ID 和 generation 必须与目标载荷逐项一致；同时移除上一主题中由 Loader 拥有但新载荷未声明的陈旧项。
- Loader 必须维护可核对的所有权记录，区分“应用前已存在”“本次新增”“本次覆写”及“应用后被其他功能改动”的类、属性和变量。切换或恢复默认时，只移除自己新增且当前值仍为自己写入值的状态；对自己覆写且当前值仍未漂移的状态恢复原值；对非自身状态或已经外部改动的状态保持不变并记录脱敏冲突。
- Loader 的损坏自修复至少覆盖：专属样式节点缺失、重复或内容哈希错误，专属主题 ID / generation 标记缺失或错误，受控变量、根类、根属性或资产 Data URI 缺失 / 错值，以及上一主题遗留的 Loader 自有状态。修复必须幂等，且只能操作带专属所有权记录的状态。
- Renderer 新文档、重连、重建和周期复核均使用同一精确收敛逻辑；修复后再次复核应返回稳定的 `unchanged` / `healthy`，不得持续改写 DOM 或制造重复监听器。
- 恢复默认主题必须删除 Loader 自有主题节点、运行时变量、Data URI、根属性、根类、主题 ID / generation 标记及持久化活动指针，同时保留 Codex 原生状态和汉化、模型、输入框、菜单、状态条等其他增强状态。

### 9. 非主题数据保护

以下内容属于严格非目标，主题中心不得读取后修改、覆盖或重置其值：

- 用户当前输入内容；
- 供应商配置；
- 模型配置；
- Base URL；
- API Key；
- Prompt；
- 其他与主题无关的管理设置。

如果实现过程中确需读取现有配置定位 Codex 目录，也必须只读使用，不得改变其业务值。
- Provider 导入、配置预览和签名更新仍属于既有供应商能力边界；主题中心不得借机改写、重存或重新签发任何凭据。
- 凭据展示必须继续保持遮罩；相关日志、错误信息和审计记录必须继续脱敏；底层凭据存储必须继续使用现有系统存储方案。

### 10. 结果提示与错误处理

- 应用非默认主题成功后，必须明确提示需要重启 Codex 才能生效。
- 恢复默认主题成功后，如同样需要重启 Codex，也必须用一致规则提示。
- 导入成功、应用成功、恢复成功、删除成功（若后续支持）必须严格基于后端真实结果返回。
- 导入失败、应用失败、恢复失败、回滚失败时，前端不得显示成功 toast、成功 banner 或“已应用”状态。
- 前端状态刷新失败时，不得把旧成功提示误当作本次成功结果。

### manifest 数据结构

每个主题包至少提供：

```json
{
  "format_version": 1,
  "id": "aurora-dark",
  "name": "Aurora Dark",
  "version": "1.0.0",
  "author": "Theme Author",
  "description": "Optional description",
  "preview": {
    "path": "preview.png",
    "kind": "codex-runtime-screenshot",
    "theme_version": "1.0.0",
    "codex_version": "1.2.3",
    "codex_build_fingerprint": "sha256:...",
    "runtime_payload_sha256": "sha256:...",
    "screenshot_sha256": "sha256:...",
    "captured_at": "2026-07-17T00:00:00Z",
    "width": 1440,
    "height": 900
  },
  "entry_style": "assets/theme.css",
  "assets": [
    "assets/theme.css",
    "assets/background.webp",
    "assets/icons/sidebar.png"
  ],
  "css_variables": {
    "--ccp-theme-surface": "#121417",
    "--ccp-theme-accent": "rgb(61 132 255)"
  },
  "root_attributes": {
    "classes": [
      "ccp-theme-aurora-dark"
    ],
    "attributes": {
      "data-ccp-theme-tone": "dark"
    }
  },
  "asset_variables": {
    "--ccp-theme-background-image": "assets/background.webp",
    "--ccp-theme-sidebar-image": "assets/icons/sidebar.png"
  },
  "supports": {
    "codex": true
  }
}
```

要求：

- manifest 只声明主题自身元数据与资源映射，不包含供应商、模型、Prompt、Base URL、API Key 或任何业务配置字段。
- manifest 不允许声明可执行脚本、命令、安装钩子、远程下载地址或要求联网解析的依赖。
- manifest 的路径必须相对主题包根目录，并在导入时解析为安全路径。
- `css_variables`、`root_attributes`、`asset_variables` 均为可选；缺失表示该主题不声明对应状态，空对象表示显式清空上一主题由 Loader 拥有的对应状态。未知子字段或错误值类型必须拒绝，不做宽松强制转换。
- `css_variables` 最多 128 项，键必须匹配 `^--ccp-theme-[a-z0-9](?:[a-z0-9-]{0,62})$`；每个值是最多 1024 UTF-8 字节且能被 CSS 解析器解析为单个声明值的字符串，所有值合计不超过 64 KiB。禁止分号逃逸、声明块、`url()`、`@import`、`expression()`、脚本协议和 `!important`；图片只能通过 `asset_variables` 声明。
- `root_attributes.classes` 最多 16 项且每项匹配 `^ccp-theme-[a-z0-9](?:[a-z0-9-]{0,62})$`；`root_attributes.attributes` 最多 32 项，键必须匹配 `^data-ccp-theme-[a-z0-9](?:[a-z0-9-]{0,62})$`，值为最多 256 UTF-8 字节且不含控制字符的字符串。禁止声明 `id`、`style`、`class` 字符串、`lang`、`dir`、`nonce`、ARIA、事件属性或任何非 CCP 命名空间属性。
- `asset_variables` 最多 64 项，键遵循 `css_variables` 的命名规则，值必须是 `assets` 中显式声明且通过类型 / 魔数 / MIME / 大小校验的本地 PNG、JPEG 或 WebP 相对路径；不得直接提供 URL、Data URI、CSS 函数或文件系统路径。
- 所有映射键在规范化后必须唯一，且不得与 Loader 保留的 `--ccp-theme-id`、`--ccp-theme-generation` 等内部键冲突；保留键清单由实现集中定义并进行契约测试。

### 同版本真实预览截图

- 主题卡片预览是验证产物，不是装饰资产。截图采集前必须将该 manifest 编译出的运行时载荷应用到通过身份校验的真实 Codex 主 Renderer，并确认至少一个目标区域的计算样式与载荷预期一致。
- “同版本”必须同时满足：截图记录的主题 ID / `theme_version` 与卡片一致，`runtime_payload_sha256` 与当前编译载荷一致，`codex_version` 与 `codex_build_fingerprint` 与当前安装并运行的 Codex 一致。任一不一致均将截图标记为 `stale`，卡片展示预览空态。
- 截图记录必须绑定脱敏采集会话、动态 CDP 端口对应的进程身份、Renderer 指纹、target 摘要、截图尺寸、采集时间和 SHA-256；主题包中仅有一张同名图片而没有可核对采集记录，不足以成为卡片预览。
- 导入时可暂存包内候选截图，但只有通过上述绑定和完整性复核后才能登记为 `verified`；未验证、损坏或旧版本截图不得显示为真实预览。
- 当前 Codex 升级、主题版本更新、运行时载荷变化或截图文件变化时，已有截图立即失效并进入“等待真实运行截图”空态，直至重新在真实 Renderer 采集。
- 默认主题遵循相同规则：先确认没有 CCP 主题状态，再从当前 Codex 构建真实截图；没有合格记录时保留默认卡片和预览空态。

### 真实主题视觉一致性与激活判定

- 主题图片不以“已打包”“已生成 Data URI”或“已写入状态文件”作为生效依据。manifest 中声明的每一个背景图、品牌图和其他 `asset_variables` / CSS 本地图片都必须真实进入通过身份校验的 Codex Renderer，并能从 CSSOM 或计算样式读取到对应资源值；资源须在 Renderer 内成功解码且具备非零固有尺寸，空字符串、`none`、无效 URL、解码失败或仅存在于未被引用的变量中均视为资源注入失败。
- 主题的品牌层、背景、主辅色和彩色状态、主要布局结构必须由实际运行时 CSS、受控根属性和资源共同实现。预览中出现但真实 Codex 中不存在的品牌栏、背景图、色彩覆盖、间距、分栏或卡片结构属于静态假预览；真实 Codex 中出现但预览遗漏的主题主视觉同样属于不一致。
- 卡片预览优先直接使用本节身份链采集的真实 Codex 截图。因卡片比例产生的裁切、缩放或脱敏派生图只能从该真实截图确定性生成，并保留源截图哈希与派生参数；任何声称“与实装逐项一致”的非原始截图表示，都必须在相同 Codex 构建、主题版本、运行时载荷和基准视口下通过视觉契约复核，不得由独立设计稿或另一套静态 HTML / CSS 生成。
- 视觉契约至少逐项记录并核对：品牌层是否存在及其位置、背景资源 SHA-256 与 `background-image` / `background-position` / `background-size`、关键色彩变量及目标区域计算色、主要区域顺序与尺寸关系、主题根类 / 根属性，以及预览裁切范围。任一必需项未实现、不可读取或与预览不一致时，不得把预览标记为 `verified`。
- 运行时状态至少区分“已选择 / 待重启”“正在验证”“当前生效（`active`）”和“失败”。只有唯一主题 style 节点、主题 ID / generation、全部必需 CSS 与资源变量、资源解码、关键计算样式和视觉契约均复核通过后，Loader 才能报告 `active`；仅写入 style 文本、根类或主题 ID 不构成激活成功。
- 任一必需图片、CSS / 资源变量或视觉契约校验失败时，诊断结果必须返回 `failed` 及脱敏差异类别，界面不得显示“当前生效”或登记新预览。若失败发生在切换或更新事务中，系统必须保留上一稳定主题并按 Mutation Journal 自动回滚；回滚也必须重新验证上一主题的唯一 style、资源和 `active` 状态。

### 本地图片与 Data URI 编译

- 后端在导入暂存区内解析 CSS 和 `asset_variables`，逐个解析相对路径、验证清单声明、文件类型、魔数 / MIME、尺寸、大小和哈希，再使用实际探测 MIME 进行 Base64 编码。
- 所有进入 Manager 卡片或 Codex Renderer 的本地 PNG、JPEG、WebP 图片都必须转换为规范 `data:<verified-mime>;base64,<payload>`；卡片预览也由后端以受控 Data URI 或等价内存字节接口提供，不暴露主题库存路径。
- CSS 中的本地图片与 WOFF2 `url(...)` 必须使用 CSS 解析器按 token 重写为 Data URI；禁止正则替换、字符串拼接、路径猜测和运行时文件读取。`asset_variables` 编译为 `url("data:...")` 形式的 CSS 变量值。
- 生成运行时载荷后必须重新解析 CSS，确认不存在本地相对 / 绝对路径、`file://`、远程 URL、未声明资源或不受支持的 Data URI，并记录不含图片正文的载荷 SHA-256、资源计数和总字节数。
- Data URI 编译是暂存验证的一部分；任何读取、解码、编码、容量或复核失败均使整个导入 / 应用事务失败，不得保留部分转换结果或退回 `file://`。

### 运行时资源结构

建议运行时至少区分三层目录：

- 主题库存目录：保存全部已拥有主题；
- 临时操作目录：保存导入 / 应用过程的临时文件；
- 当前应用目录或当前主题指针：保存“当前生效主题”的受控引用。

推荐语义如下：

```text
~/.claude-codex-pro/
  themes/
    registry.json
    library/
      default-baseline/
      aurora-dark/
    previews/
      <theme-id>/<theme-version>/<codex-build>/
    tmp/
      operations/<operation-id>/
    active/
      current-theme.json
      loader-ownership.json
      assets/...
```

具体路径可按现有项目状态目录规范调整，但必须满足隔离、可备份、可回滚、可清理。

    },

    },

- Journal 用于借鉴 Codex-X 的事务 / Mutation Journal 思路，记录每次导入、应用、恢复默认和回滚。
- Journal 必须支持崩溃恢复：若上次事务停留在未提交状态，系统下次启动时必须能够识别并继续回滚或完成清理。
- Journal 不得写入明文凭据、供应商密钥、Prompt 或用户输入内容。

### Renderer 身份校验与截图验收

- 主题应用成功不应仅以文件层写入为准，还必须具备借鉴 Dream Skin macOS 的目标 Renderer 验证边界。
- 若实现依赖 CDP 检查主题落地效果，必须先验证目标连接确属 Codex Renderer，而不是其他 Chromium 页面或外部 WebView。
- CDP 验证至少应覆盖：
  - 动态端口发现；
  - 目标页面身份校验；
  - Renderer 指纹确认；
  - 主题注入后的真实 Codex Renderer 截图验收。
- Renderer 指纹应基于稳定的目标特征，如应用标识、预期根容器、注入桥能力或受控版本标记，不得依赖脆弱的随机 DOM 扫描。
- 真实截图验收用于证明主题视觉层已生效，且截图必须来自真实 Codex Renderer，不得使用静态示意图冒充。
- 用作主题卡片预览的截图必须来自本节同一身份校验链，并通过主题 ID / 版本、Codex 版本 / 构建指纹、运行时载荷 SHA-256 与截图 SHA-256 四重绑定；验收截图与卡片实际读取的预览文件必须是同一哈希对象。
- 截图采集必须避开凭据、用户输入正文和不必要的本地路径；采集后按主题版本和 Codex 构建隔离保存，不得用新版本记录覆盖旧版本后继续沿用旧哈希。

### 启动与重启注入契约

- 启动器必须将“当前存在非默认主题”视为独立的前端注入需求；即使翻译、模型标识、记忆注入及其他增强项均关闭，也必须启动 helper、等待 Codex CDP 并安装主题新文档脚本。
- 默认主题本身不应无条件开启注入；未启用其他前端增强时，继续允许跳过 helper 和注入，避免改变既有轻量启动行为。
- 已有 Launcher 实例收到再次启动请求，以及 Manager 执行“启动 / 重启 Codex”时，必须使用同一主题判定，不得出现首次启动生效、重启失效或相反的分叉。
- Windows 打包版启动前必须验证请求的 CDP 端口是否可绑定；请求端口被占用或处于系统僵尸监听状态时，必须为本次启动选择非零动态端口，并将该实际端口贯穿 Codex 启动参数、Launcher 状态、helper、注入和 watchdog。
- Manager 的重启监控必须以本次重启开始时间为边界，接受 Launcher 在该边界之后报告的实际动态端口，并拒绝此前遗留的启动状态；不得因实际端口与最初请求端口不同而误报注入超时。
- 动态端口分配返回无效端口时不得向 Codex 传入端口 `0`；应保留请求端口并通过既有诊断链记录后续上线失败，确保错误可定位且状态不被伪造为成功。
- 重启后的诊断记录至少包含当前主题 ID、generation、debug/helper 端口、CDP/Renderer 等待阶段，以及最终是否观察到 Renderer 注入心跳；主题读取失败时记录脱敏阶段，不输出主题 CSS 全文或用户路径。
- 主题脚本仍作为独立的新文档脚本注册，不得拼入汉化、模型标识、输入框或菜单处理脚本。
- Renderer 注入成功至少需要同时满足：专属主题 `<style>` 唯一存在、所需根类已挂载、主题 ID/generation 标记一致，且目标区域计算样式发生预期变化；不能只依据脚本执行成功或样式文本已写入判定视觉生效。
- 新文档脚本必须调用同一 Theme Loader 精确收敛入口，对样式节点、`css_variables`、`root_attributes`、`asset_variables`、Data URI 和所有权记录逐项复核；发现 Loader 自有状态损坏时就地修复，并在修复后再次验证载荷哈希。
- Loader 的诊断结果至少区分 `applied`、`repaired`、`healthy`、`ownership_conflict` 和 `failed`，并列出脱敏的差异类别，不记录 CSS / Data URI 正文；`repaired` 只有在修复后完整复核通过时才算注入成功。

### IPC 与前后端交互

- 可以新增主题中心所需的新命令，但不得修改既有 Tauri IPC 的名称、参数、返回结构和既有调用时机。
- 新增主题相关 IPC 必须遵循现有 `ok` / `failed` 风格返回约定，前端据此决定成功 / 失败提示。
- 主题列表读取、导入、应用、恢复默认、删除（若后续支持）都应返回可审计的结果字段，如 `theme_id`、`applied`、`rolled_back`、`restart_required`、`message`。
- 如包含 Renderer 验证，返回结构应能表达 `renderer_verified`、`renderer_fingerprint`、`preview_status`、`theme_version`、`codex_version`、`runtime_payload_sha256`、`screenshot_sha256` 或等价字段；不得向前端返回截图绝对路径，且不得破坏既有 IPC 约定。
- 前端不得根据按钮点击本身推断成功，必须以后端结果为准。

## 技术约束

- 保持现有 Rust + Tauri + React + Vite 工作区架构。
- 保持现有业务、既有 Tauri IPC、非主题页面和非主题状态行为不变。
- 不回滚、不覆盖其他代理正在进行的改动。
- 仅在主题中心相关范围内新增前端路由、页面组件、后端命令、核心主题服务、状态存储与测试。
- 主题应用链路必须使用可验证的文件操作策略：安全解包、基线快照、原子写入、失败回滚、临时目录清理。
- manifest 解析必须采用拒绝未知字段的结构化反序列化和 CSS / 图片 / 字体对应的真实解析器；路径、名称、类型、大小与计数限制均在后端执行，前端检查仅用于即时反馈。
- 图片 Data URI 转换和 Loader 运行时载荷编译必须具备确定性：相同规范化 manifest 与资产字节产生相同 SHA-256，字段遍历顺序、MIME 选择和 Base64 编码不得随平台变化。
- 主题资源写入与回滚测试必须参考 Codex-X 的事务化思路：可检测未完成变更、可验证回滚结果、可保留上一版本。
- 对任何真实文件替换都必须优先保证“可恢复性”高于“应用速度”。
- 不引入以修改用户 Codex 业务配置为代价的捷径实现。
- 任何需要与 Codex 注入资源协作的实现，都必须通过独立主题挂载层完成，不得直接把主题逻辑混入汉化、模型、输入框或菜单注入逻辑中。
- Theme Loader 的所有权模型、损坏自修复和仅恢复自身状态属于持久化契约，不得以全量清空 `documentElement.className`、删除所有 `data-*`、重置内联样式或重建根节点实现。
- 若使用 CDP / Renderer 验证，必须参考 Dream Skin macOS 的身份校验思路，确保动态端口、目标身份、Renderer 指纹与真实截图链路完整可信。

## 交付范围

本规格对应的后续实现至少应覆盖：

- 独立一级导航“主题中心”；
- 页面顶部“新建主题”主操作；
- 默认展示全部已拥有主题的主题列表页；
- 三列稳定卡片 + 窄屏响应布局；
- 永久置顶的 Codex 默认主题卡片与真实恢复链路；
- 主题清单持久化结构；
- 事务日志 / Mutation Journal；
- manifest、预览图与样式资源规范；
- 受控 `css_variables`、`root_attributes`、`asset_variables` schema 与精确 Loader；
- 同主题版本、同 Codex 构建的真实运行截图预览、版本失效规则和默认 / 待采集空态；
- 本地图片类型 / 尺寸 / 大小验证、Data URI 确定性编译与无本地路径运行时载荷；
- 导入校验；
- 临时目录验证、原子替换、上一版本保留、回滚与失败提示；
- 占用检测、恢复测试与崩溃恢复策略；
- CDP 身份校验、动态端口、Renderer 指纹与真实截图验收边界；
- Loader 自有状态损坏修复、所有权记录和只恢复自身状态的机制；
- 与既有汉化 / 模型 / 输入框 / 菜单注入隔离的应用机制；
- Provider 导入 / 配置预览 / 签名更新仅作为既有边界、凭据保持遮罩与脱敏的约束；
- 不修改用户输入、供应商、模型、Base URL、API Key、Prompt 的保护约束；
- 对应测试、文档与验收材料。

## 验收关注点（供后续 acceptance 使用）

- 主题中心是否为独立一级导航。
- 顶部是否存在“新建主题”。
- 首屏是否默认展示全部已拥有主题。
- 桌面端是否稳定为每行三张图片主导卡片，窄屏是否正确响应。
- 卡片预览是否来自同主题版本、同 Codex 构建、同运行时载荷的真实 Renderer 截图，缺失 / 过期时是否显示空态而非替代图。
- 主题图片是否真实进入 Codex Renderer 并完成解码，品牌层、背景、色彩和布局是否与卡片预览逐项一致，而不是只存在于静态预览。
- 仅有默认主题时是否同时保留首位默认卡片和明确导入空态。
- 第一张是否永久为 Codex 默认主题，且能真实恢复原貌。
- 应用非默认主题后是否提示重启 Codex。
- 导入 / 应用失败时是否绝不显示成功。
- manifest 是否受控支持 `css_variables`、`root_attributes`、`asset_variables`，主题导入是否具备严格路径 / 命名 / 类型 / 魔数 / MIME / 文件数 / 大小校验、临时目录验证、原子替换、上一版本保留与回滚。
- 本地图片是否全部经验证后转换为 Data URI，运行时是否不存在主题库存路径、`file://` 或远程资源回退。
- 是否具备事务日志 / Mutation Journal、占用检测与恢复测试。
- Loader 是否精确应用 manifest、修复自身损坏并只恢复自身拥有的状态。
- Loader 是否仅在 CSS / 资源变量、唯一 style 节点、计算样式和视觉契约全部通过后报告 `active`，失败时是否明确报错并恢复上一稳定主题。
- 主题应用是否与汉化、模型、输入框、菜单注入隔离，且 Renderer 注入为独立视觉层。
- 是否具备 CDP 身份校验、动态端口、Renderer 指纹与真实截图验收边界。
- 是否完全不修改用户输入、供应商、模型、Base URL、API Key、Prompt。
- 是否保持 Provider 导入、配置预览、签名更新仅为既有边界，且凭据继续遮罩、脱敏并保留系统存储。
- 是否保留现有业务、既有 Tauri IPC 与非主题行为。

## GitHub 精选主题包

仓库可在 `Theme/` 中维护经过审计的社区主题适配包，作为用户可直接导入的本地主题样本。首批精选包必须满足以下要求：

- 每个主题同时提供可直接浏览的目录包与同名 ZIP 包；ZIP 根目录直接包含 `theme.json`，不得额外嵌套仓库名或版本目录。
- `Theme/README.md` 必须记录主题 ID、上游仓库 URL、固定提交 SHA、上游许可证、素材许可证和适配说明，保证来源可追溯。
- 仅提取获得明确再分发许可的背景图、真实应用截图和视觉参数；不得带入上游 Renderer 脚本、安装脚本、CDP 控制代码、远程加载代码或可执行文件。
- 适配 CSS 必须由 CCP Theme Loader 独立加载，所有选择器都以该主题的 `ccp-theme-*` 根类限定范围；不得复用上游 DOM 标记脚本，也不得写入输入内容、菜单项、模型名称或汉化节点。
- 主题卡片预览优先采用上游公开的同主题真实 Codex 新任务截图。来源仅提供设计稿时，该主题不得进入本批精选包。
- 每个包必须包含 `LICENSE` 与 `NOTICE.md`，清楚区分适配代码、背景素材和 Codex 产品截图各自的权利与归属。
- 单个目录包和 ZIP 包都必须通过当前 `CodexThemeStore` 导入校验；目录包与 ZIP 包导入后得到的主题 ID、版本和运行时资源应一致。
- 精选主题属于可选本地资产，不在安装或启动时自动应用，不改变用户当前主题与任何非主题配置。
