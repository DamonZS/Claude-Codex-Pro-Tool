# 盘古记忆当前对话候选修复

## 背景

用户要求确认盘古记忆是否记录了当前这条对话。实际检查发现本地 `memory_assist.sqlite` 中长期记忆与待确认候选均为 0，日志中只有 `renderer.memory_runtime` 与 `/memory/session` 心跳，没有最近的 `/memory/candidates` 请求。

## 目标

本次要完成：

- 明确诊断当前对话没有进入盘古记忆的原因。
- 让 Codex 注入脚本能够识别“盘古记忆是否记住这条对话、没有就找原因修复”这类记忆自检/修复指令，并对高置信内容自动写入长期记忆。
- 在自动学习未触发时记录限流诊断，便于区分没有用户消息、文本不可学习、重复记忆和写入失败。
- 保留待确认候选能力，但高置信自动学习路径不再先进入候选。

本次不包含：

- 执行或解密左脑黑箱算法。
- 清空、迁移或重建用户现有记忆数据库。
- 修改 Claude 中文注入脚本。

## 用户视角描述

用户在 Codex 对话中输入与盘古记忆自身有关的明确检查或修复指令后，盘古记忆应把高置信指令自动写入长期记忆。若没有触发学习，日志应显示被哪一道条件拦住，而不是只显示 0 条记录。

## 功能要求

- 自动候选识别应覆盖中文“盘古记忆/记忆 + 是否/有没有/没有/原因/修复/记录/候选/监听/对话/会话”。
- 自动候选识别应覆盖“这条对话/当前对话/本条对话 + 记忆/记录/候选/盘古”。
- 自动候选识别应覆盖英文 `pangu` 或 `memory` 与 `candidate/record/remember/debug/fix/session` 组合。
- 自动候选应优先读取 `data-message-author-role="user"` 等显式用户消息；当新版 Codex DOM 只暴露右侧用户气泡结构时，允许复用已限定的 Codex 用户气泡识别逻辑作为用户消息来源。
- 用户消息 fallback 必须限定在 conversation root 内，不得从侧边栏、标题、导航、通用 `main [class*="user"]` 或任意 conversation turn 猜测用户内容。
- 自动候选未触发时，注入脚本应发送不包含原文的诊断事件，记录原因和文本长度。
- 诊断事件必须限流，避免在 DOM 扫描循环中刷屏。

## 技术约束

- 不新增依赖。
- 不改变后端记忆数据库 schema。
- 不修改 `assets/inject/claude-chinese-inject.js`。
- 不删除或重置 `~/.claude-codex-pro/memory_assist.sqlite`。
