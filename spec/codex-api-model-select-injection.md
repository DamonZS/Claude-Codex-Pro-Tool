# Codex API 模型选择注入

## 背景

Codex 在 API 中转或兼容模式下可能只显示内置模型。供应商 `/v1/models` 已返回的自定义模型虽然可以用于请求，但未必出现在 Codex 模型选择器中。CCP 通过本地模型目录、应用状态补丁和有限的 DOM 兜底，让这些模型可见、可选且能写入后续请求。

当前缺陷是 DOM 兜底把 Radix 外层包装、普通菜单和页面区域误判成模型菜单，导致每个区域都追加一组模型名称并覆盖正常界面。

## 目标

- 保留 `modelWhitelistUnlock` 控制的模型解锁能力。
- 使用 CCP 本地 `/codex-model-catalog` 提供已配置或发现的模型列表。
- 优先修补 Statsig、React 状态和模型协议结果，DOM 注入只作为兜底。
- DOM 模型项只能出现在真实 Codex 模型选择菜单中。
- 重复扫描保持幂等，并能清理旧版本遗留在错误容器中的注入组。

## 非目标

- 不修改供应商配置、生产 Base URL 或鉴权材料。
- 不伪造供应商未返回且用户未配置的模型。
- 不重做 Codex 原生模型选择器 UI。
- 不改变权限、推理强度、项目、设置等其他菜单行为。

## 功能要求

- 模型目录继续由 `/codex-model-catalog` 提供，模型条目保留 ID、显示名、上下文窗口和推理能力等已有描述。
- 只有菜单自身拥有的原生模型项可以作为识别证据；嵌套菜单中的模型文本不得让父级 portal、dialog 或页面容器通过识别。
- 原生模型项的归属由其最近的模型菜单表面确定，不能用不受约束的后代文本扫描。
- 普通菜单、权限菜单、推理强度菜单、项目菜单、CCP 自身弹窗和页面根容器不得接收模型注入组。
- 每个合法模型菜单最多存在一个 CCP 模型组，每个模型最多存在一个注入项。
- 每轮扫描先识别合法模型表面，再删除父节点不属于合法表面的旧注入组。
- 模型菜单关闭或 DOM 结构变化后，不得在页面中留下游离模型项。
- 关闭模型解锁时应移除 DOM 兜底注入，不影响原生模型项。

## UI 与交互

- 注入项沿用当前紧凑模型菜单样式，不新增弹窗或常驻面板。
- 注入项显示真实模型 ID，并以 `CCP 模型增强` 标识来源。
- 点击后继续写入当前模型选择、更新请求覆盖并关闭当前模型菜单。
- 不改变其他 Codex 页面、导航、火焰效果或管理工具 UI。

## 数据与安全

- 不在 DOM、日志、诊断和测试中输出 API Key、Bearer token、完整 Base URL 或私密会话内容。
- 模型候选识别只读取当前可见菜单结构和模型名称，不读取用户对话正文。

## 交付范围

- `assets/inject/renderer-inject.js` 的模型菜单识别、清理与幂等逻辑。
- `crates/claude-codex-pro-core/tests/cdp_bridge.rs` 的 DOM 契约回归测试。
- 本规格及对应验收文档。
