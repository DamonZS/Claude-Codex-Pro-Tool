# ccswitch 重复导入修复旧供应商记录

## 背景

此前从 ccswitch 导入的供应商可能已经以空 key 状态保存到管理工具设置中。后续修复虽然能让新的导入结果带上 key，但前端遇到同 ID 时会生成重复供应商，而不是替换旧的 ccswitch 记录，导致用户点击“使用”仍可能点到旧空记录。

## 目标

本次要完成：

- 再次从 ccswitch 导入同 ID 供应商时，如果本地已有记录也是 ccswitch 来源，应替换旧记录。
- 替换后保留原 ID，使当前激活项和用户点击目标都指向已修复记录。
- 如果同 ID 本地记录不是 ccswitch 来源，不覆盖用户手工供应商，仍使用唯一 ID 追加。

本次不包含：

- 不暴露真实 API Key 到日志、测试输出或文档。
- 不改供应商页面整体 UI。
- 不改变后端不持久化明文 `apiKey` 字段的安全设计。

## 功能要求

- ccswitch 导入前端合并逻辑必须识别 ccswitch 来源记录。
- 同 ID 且旧记录为 ccswitch 来源时执行替换。
- 同 ID 且旧记录非 ccswitch 来源时执行唯一 ID 追加。
- 导入完成提示应区分新增和更新数量。

## 验证要求

- 添加或更新源码断言测试，证明替换逻辑存在。
- 前端类型检查必须通过。
