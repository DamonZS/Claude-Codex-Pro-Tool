# 修复 Claude 入口

## 背景

用户在 Claude 官方客户端白屏后，已通过现有的 Claude 官方文件还原链路恢复正常。当前管理工具中该还原入口位于 Claude 中文相关区域，不够符合用户处理故障时的使用路径。用户希望把这个已验证有效的能力集成到「诊断与修复」卡片中，并命名为「修复 Claude」。

## 目标

本次要完成：

- 在管理工具概览页「诊断与修复」卡片新增按钮「修复 Claude」。
- 点击按钮时复用现有 `restoreClaudeZhPatch()` 前端 action。
- 继续调用既有 Tauri 命令 `restore_claude_zh_patch`，复用现有还原、关闭 Claude、恢复 locale 与反馈逻辑。
- 给测试增加锚点，防止后续 UI 调整再次移除该入口。

本次不包含：

- 不新增新的 Claude 还原算法。
- 不删除 Claude 账号、缓存、数据库或用户数据。
- 不触碰 Codex 启动、注入或修复逻辑。
- 不改动 Claude 中文注入/汉化安装逻辑。
- 不做无关 UI 重构。

## 用户视角描述

用户打开管理工具概览页，在「诊断与修复」区域看到「修复 Claude」按钮。点击后，管理工具显示现有恢复 Claude 官方文件的确认与运行反馈，并执行既有还原链路。完成后展示后端返回结果，用户可据此确认 Claude 是否恢复。

## 功能要求

- 「诊断与修复」卡片必须展示「修复 Claude」按钮。
- 按钮点击必须调用 `actions.restoreClaudeZhPatch()`。
- 该 action 必须继续调用 `restore_claude_zh_patch` Tauri 命令。
- 仍保留现有确认弹窗、运行中提示、结果提示和状态刷新行为。
- 失败时沿用现有错误处理，不吞掉错误。

## UI / 交互要求

- 按钮文案为「修复 Claude」。
- 按钮放在「诊断与修复」卡片中，与「刷新概览」「刷新 Claude 第三方配置」「修复前端连接」「修复后端服务」同级。
- 不改变该卡片现有布局结构和其它按钮。

## 数据与接口要求

- 前端只复用现有 action，不新增 Tauri 命令。
- 后端继续使用 `restore_claude_zh_patch`。
- 不新增数据库表，不改数据存储位置。

## 技术约束

- 遵守最小改动原则。
- 不重写还原逻辑。
- 不删除或重置用户数据。
- 不杀 Codex 进程。
- 如构建时 `claude-codex-pro-manager.exe` 被占用，只允许终止 manager 进程。

## 交付范围

- 规格文档。
- 验收标准文档。
- 前端按钮接入。
- 前端/结构测试锚点更新。
- 定向验证与 manager debug 构建。
