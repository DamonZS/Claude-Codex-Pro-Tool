# 盘古记忆 0 条记录修复

## 背景

用户反馈 Codex 注入标识和管理工具都显示盘古记忆已启用、已注入、对话监控运行中，但长期记忆和待确认记录一直为 0。诊断日志显示运行中的旧注入脚本仍上报 `workspace: "codex:你好"`，说明旧脚本仍可能被加载；同时当前自动候选生成只识别少量“记住/以后/项目约定”触发词，普通但有价值的项目指令不会进入待确认列表。

## 目标

本次要完成：

- Codex 注入脚本不再把会话标题作为盘古记忆 workspace。
- 自动候选生成覆盖真实用户消息中的项目约束、偏好、修复要求和 UI/工作流要求。
- 普通寒暄、过短消息、纯标题不得生成候选记忆。
- 生成候选后必须用后端返回的真实 `pendingCandidates` 更新注入标识。
- 重新构建 debug 启动器和管理工具，确保本地调试 exe 使用最新注入脚本。

本次不包含：

- 自动把候选写入长期记忆。
- 删除或重置用户现有记忆数据库。
- 修改 Claude 中文注入脚本。

## 用户视角描述

用户在 Codex 中输入有明确项目规则、偏好或修复要求的消息后，盘古记忆应生成待确认候选。用户确认后才进入长期记忆。输入“你好”等寒暄时，不应产生候选。

## 功能要求

- 自动候选仍必须只读取明确用户角色消息。
- 没有真实用户消息时，不得调用 `/memory/session`。
- `codexMemoryWorkspace()` 只能使用项目路径、项目 id、线程 id 或路径哈希，不得使用 `document.title`、侧栏标题或当前对话标题。
- 候选创建成功后应刷新或采用后端真实计数，避免前端本地 `+1` 与后端去重逻辑不一致。
- 自动候选创建失败不得污染长期记忆，也不得记录敏感原文到普通日志。

## 技术约束

- 保持现有 React/Tauri/Rust 架构。
- 不新增依赖。
- 不修改 `assets/inject/claude-chinese-inject.js`。
- 不删除或重置 `~/.claude-codex-pro/memory_assist.sqlite`。

## 交付范围

- `assets/inject/renderer-inject.js`
- 相关 Rust 测试
- 本规格和对应验收标准
- 重新构建后的本地 debug 可执行文件
