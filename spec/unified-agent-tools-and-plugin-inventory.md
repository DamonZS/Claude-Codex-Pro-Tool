# Claude、Codex 工具与插件统一资产清单

## 背景

“工具与插件”页面当前把 Codex 与 Claude 分成两套面板，并主要从受管配置或少量硬编码状态生成条目。真实环境中已经存在大量 MCP、Skills 与插件，但页面只显示少数条目；仓库卡还会把“配置文本已写入”表述为“已解锁/已注册”，无法证明应用实际可见。

本任务参考 cc-switch 的统一资产交互：同一资产只显示一行，行尾的 Claude、Codex 应用图标分别表示该资产在对应应用中的真实启用状态。点击图标只改变对应应用，不影响另一端。

## 目标

本次包含：

- 将 Codex、Claude 的 MCP、Skills、插件合并到一个标题为“Claude、Codex 工具与插件”的面板。
- 完整发现本机可解析的 Codex 与 Claude 资产，而不是只显示本工具受管条目。
- 同一资产跨应用去重合并，并保留每个应用独立的启用状态和来源路径。
- 以 Claude、Codex 应用图标作为独立开关：点亮表示该端可用，点灭表示该端关闭。
- 修复“检测”按钮无反馈，提供检测中、成功、失败和数量摘要。
- 删除“同步到当前 Codex”按钮。
- 将插件仓库状态拆分为配置写入、本地来源和应用可见性，禁止用“已解锁”笼统代替真实状态。

本次不包含：

- 自动执行未知第三方仓库脚本或 hooks。
- 自动信任、安装用户未选择的第三方资产。
- 删除 Skills 或插件的唯一原始副本。
- 重置 Codex、Claude 配置或其他用户数据。
- 修改供应商、盘古记忆、Claude 汉化注入或发布流程。
- 增加 Gemini、OpenCode、Hermes 等其他应用入口。

## 用户视角描述

用户进入“工具与插件”页面后，顶部仍可审查 Codex 与 Claude 插件仓库的真实状态。下方只出现一个“Claude、Codex 工具与插件”面板，可以在 MCP、Skills、插件三类资产之间切换。

每项资产只占一行，显示名称、摘要、来源，以及 Claude 和 Codex 图标。图标点亮表示该资产已在对应应用中启用；点击亮图标只关闭该应用中的资产，点击灰图标只把资产启用到该应用。操作必须显示处理中及最终结果，并在完成后重新检测真实状态。

点击“重新检测”后，按钮立即进入加载态；完成时显示 Codex/Claude 各类别数量与扫描路径摘要，失败时显示可读错误。页面不再提供“同步到当前 Codex”。

## 功能要求

### 1. 统一资产模型

统一资产至少包含：

- 稳定 ID。
- 类型：`mcp`、`skill` 或 `plugin`。
- 展示名称与可选摘要。
- 一个可审查的主来源路径或配置来源。
- Claude 状态：已启用、未启用、是否可切换。
- Codex 状态：已启用、未启用、是否可切换。
- 对应应用的配置体或可恢复源路径，用于执行切换。

同类型且标准化 ID 相同的资产必须合并成一行；不同类型不得误合并。合并不得丢失任一应用的状态。

### 2. Codex 完整发现

Codex MCP 必须读取当前 `CODEX_HOME/config.toml` 中所有可解析的 MCP 表，包括受管与非受管条目。

Codex Skills 必须发现当前 `CODEX_HOME/skills` 下所有包含 `SKILL.md` 的有效技能目录。允许发现嵌套技能，但同一路径不得重复计数。

Codex 插件必须聚合当前配置中的插件条目、已安装记录和本地插件来源。仅存在于仓库缓存、尚未启用到 Codex 的插件可作为可用来源，但 Codex 图标不得因此误点亮。

Codex 前端注入清单必须读取所有已配置且本地快照真实存在的 marketplace，包括 OpenAI、第三方、Product Design 和用户自定义本地仓库。每个来源必须从可读的 `marketplace.json` 展开；只有配置、没有本地快照时不得注入，也不得报告为“应用可见”。

### 3. Claude 完整发现

Claude MCP 必须聚合 Claude Code 与 Claude Desktop 当前可读配置路径中的 `mcpServers`，包括 MSIX 活跃配置路径和 `projects.*.mcpServers`；同一 MCP 按 ID 合并。关闭某个 Claude MCP 时，必须清理所有已发现配置路径及项目级嵌套中的同 ID 条目，重新检测后不得因残留副本再次点亮。

Claude Skills 必须发现 `~/.claude/skills` 下所有包含 `SKILL.md` 的有效技能目录。

Claude 插件必须聚合 `~/.claude/plugins` 的已安装/启用记录和 Claude Desktop 可读插件状态。只存在缓存的插件不得误标为已启用。

单个不存在、损坏或不可读的候选路径不得使整个检测失败；结果必须携带可读诊断摘要。

### 4. 应用图标切换

- MCP：通过对应应用支持的结构化配置写入改变启用状态，写入前沿用项目现有备份和原子写入能力。
- MCP：Claude 同一 MCP 跨多个配置文件更新时，任一文件写入失败必须恢复已尝试文件的原始内容，不得留下部分关闭状态。
- Skills：启用时从已有可审查来源同步到目标应用技能目录；关闭时移动到本工具管理的可恢复停用位置或采用应用支持的禁用机制，不得物理删除唯一副本。嵌套 Skill 必须保留相对目录层级。
- 插件：优先使用应用原生启用配置；若只能通过本地目录生效，必须采用可恢复停用方式。缓存插件首次启用时必须使用配置或 marketplace manifest 中的真实 marketplace ID，不得使用缓存目录名伪造 ID。
- 切换 Claude 不得改变 Codex，切换 Codex 不得改变 Claude。
- 切换失败时前端状态必须回滚为重新检测后的真实值，并显示失败原因。

### 5. 仓库状态

Codex 和 Claude 仓库状态必须分别表达：

- 配置是否写入。
- 本地仓库/快照来源是否存在并可读。
- 应用是否已检测到或是否需要重启后确认。

只有配置写入时不得显示“已解锁”“应用已可见”或等价结论。无法验证运行时可见性时，必须显示“待应用确认”或“重启后确认”。

Claude 状态必须聚合所有候选配置路径并标明当前实际写入/发现路径，不能只读取第一个默认路径。

### 6. 检测反馈与日志

- 页面只保留一个“重新检测”入口。
- 点击后按钮立即进入加载态并禁止重复触发。
- 成功提示至少包含 MCP、Skills、插件总数以及 Claude/Codex 启用数量。
- 失败提示必须包含失败阶段，但不得包含 API key、token、cookie 或完整敏感配置。
- 检测和切换操作必须写入现有管理工具日志，记录动作、资产 ID、目标应用和结果，不记录配置密钥原文。

## UI / 交互要求

- 仓库状态卡位于统一资产面板上方，可保留 Codex、Claude 两张卡，但文案必须使用真实三态。
- 统一面板只使用一个标题“Claude、Codex 工具与插件”。
- 面板包含 MCP、Skills、插件三个标签及真实数量。
- 每条资产使用紧凑列表行，不再为 Codex/Claude 分别重复展示。
- 行尾固定展示 Claude、Codex 两个可识别应用图标；亮色表示启用，灰色表示未启用。
- 图标必须有可访问名称、悬停提示、加载态和禁用态。
- 编辑、删除等高级操作只在对应资产确实支持时显示，不得遮挡应用图标。
- 空态需要说明本次扫描过的来源与未发现原因。
- 统一 MCP 标签内必须保留“新增 MCP”入口；用户可选择 Claude 或 Codex 作为目标应用，保存成功后重新检测统一清单。不得为了恢复新增能力重新引入两套分离面板。
- 删除“同步到当前 Codex”按钮及相关用户文案。

## 数据与接口要求

后端提供一个统一检测接口，返回统一资产列表、分类计数、各应用计数、扫描路径与非致命诊断。接口不得把密钥或完整敏感配置返回给仅用于列表展示的前端。

后端提供一个按资产、目标应用和目标状态切换的接口。请求至少包含资产稳定 ID、类型、目标应用和 `enabled`；后端必须依据重新发现的资产来源执行，不能信任前端传入任意文件路径。

现有 Codex/Claude 单端接口可以为兼容性保留，但统一页面不得依赖两个彼此独立、语义不一致的列表拼接。

## 技术约束

- 优先把扫描、标准化、合并和切换放入 core 层，Tauri commands 只负责异步边界与结果包装。
- 不引入新的前端拖拽或状态管理依赖。
- 文件遍历必须有明确根目录、符号链接循环保护和合理深度/数量上限。
- 所有配置写入沿用现有备份、结构化解析和原子写入能力。
- Windows 下由管理器自动触发的 Git、CLI 或其他后台子进程必须使用无控制台窗口启动标志；打开管理器、检测仓库或建立 marketplace 本地快照时不得弹出 Windows Terminal、cmd 或 PowerShell 窗口。
- 不记录敏感配置原文。
- 只修改本任务相关代码与文档，不回滚工作区其他改动。

## 交付范围

- 本规格与对应验收文档。
- core 层统一资产发现、合并与切换能力及 Rust 测试。
- Tauri 统一检测/切换命令和命令注册。
- React 统一资产面板、图标开关和检测反馈。
- 仓库状态文案与判定修复。
- 前端类型检查、构建、定向 Rust 测试和 manager 构建证据。
