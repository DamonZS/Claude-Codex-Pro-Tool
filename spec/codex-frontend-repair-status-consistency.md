# Codex 前端修复状态一致性

## 背景

Codex 已通过新版启动器完成重启和脚本注入，GPT-5.6 模型也能正常选择与调用，但管理工具“修复前端连接”仍提示“未确认可注入”。现场日志证明 Codex DevTools `/json`、helper 和 renderer 心跳均已正常；最小复现进一步证明 Chromium 已返回完整 HTTP 正文，却没有立即关闭连接，现有探测器因 `read_to_string` 最终超时而误判离线。

## 目标

- 修复 Codex CDP `/json` 在线探测的假阴性。
- 让“修复前端连接”的最终提示与概览页真实在线状态一致。
- 为“响应完整但连接保持打开”的 Chromium 行为增加回归测试。

## 非目标

- 不修改 GPT-5.6 模型目录、模型请求覆盖和注入菜单。
- 不修改用户 `config.toml`。
- 不重置盘古记忆数据库或其他用户数据。
- 不为了验证而重启当前 Codex 会话。

## 功能要求

- Codex DevTools `/json` 返回 `HTTP 200` 且已收到 `webSocketDebuggerUrl` 时必须判定在线。
- 探测不得要求服务端主动关闭 HTTP 连接。
- 响应不完整、非 `HTTP 200` 或不含 `webSocketDebuggerUrl` 时仍必须判定离线。
- 修复命令继续要求真实 Codex 重启与新注入，不得重新接受修复前的旧心跳。

## 技术约束

- 最小修改 `apps/claude-codex-pro-manager/src-tauri/src/commands.rs`。
- 不引入新依赖。
- 保留现有命令名、前端按钮和返回结构。

## 交付范围

- CDP HTTP 探测修复。
- Rust 回归测试。
- 前端检查与 Manager 重建证据。

## 补充要求：强制重启必须避开未释放的 CDP 端口

- “修复前端连接”结束旧 Codex 后，必须等待首选 CDP 端口可以重新绑定，不能只以旧 PID 消失作为端口已释放的依据。
- 首选端口在限定时间内仍不可绑定时，本次强制重启必须选择一个可用的回环备用端口。
- 启动请求、状态文件、上线等待和后续注入必须使用同一个实际 CDP 端口。
- 状态文件仍指向旧端口时，上线等待必须继续直接探测本次请求端口；旧状态不得阻断新端口确认，也不得冒充本次启动成功。
- 备用端口只用于 Manager 的强制重启修复链，不修改 Windows packaged Codex 的全局端口复用策略。

## 补充要求：注入心跳必须属于当前启动代际

- Manager 只有在存在当前 Codex 启动状态，且 renderer 心跳时间不早于该状态的 `started_at_ms` 时，才能把心跳判定为当前注入证据。
- 仅满足 45 秒新鲜度、但早于当前启动时间的旧心跳不得显示“Codex 已注入”或“前端在线”。
- 当前启动之后产生的有效心跳仍按现有新鲜度和运行时状态规则判定。

## 补充要求：注入控制台连接状态不得被慢请求拖垮

- CDP bridge 必须持续读取 `Runtime.bindingCalled` 事件；某个慢业务路由执行期间，后续 `/backend/status` 请求仍须被接收和处理。
- bridge 处理结果允许按实际完成顺序回写，不得为了保持请求进入顺序而阻塞健康检查。
- 重装 bridge 脚本时必须保留尚未完成的回调表和递增请求序号，避免在途请求因重注入永久挂起。
- renderer 脚本每次重注入必须替换旧心跳定时器，并使用当前脚本代际阻止旧异步检查覆盖新状态。
- 单次短暂超时或失败不得立即把最近一次成功连接显示为“未连接”；连续失败达到阈值后才显示离线，成功检查必须立即恢复并重置失败计数。
- `/backend/status` 必须同时探测 bridge 与 helper，任一通道先成功即保持在线；不得等 bridge 超时后才启动 helper。`/backend/repair` 仍按顺序执行，避免重复触发修复副作用。
- 页面从后台恢复可见时必须立即重新检查，避免 Chromium 后台定时器节流导致红色状态长期不恢复。
- watchdog 与 renderer 心跳不得保持相同的固定周期，减少健康检查同频争用；不改变现有 UI 布局和连接状态文案。
