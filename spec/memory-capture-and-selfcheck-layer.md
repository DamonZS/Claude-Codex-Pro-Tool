# 盘古记忆会话采集与自检增强

## 背景

当前盘古记忆在 Codex 页面中能显示注入标识和 runtime 心跳，但用户反馈长期记忆、待确认候选和记录数长期为 0。已有日志能看到 `/memory/session` 请求和 `renderer.memory_runtime`，却无法稳定说明“真实用户消息是否被捕获、为什么没有生成候选、候选是否写入数据库、manager 状态是否与 runtime 一致”。

本次可取的工程思想是：会话启动时加载摘要、按 workspace 隔离、自动学习先进入待确认、注入与自检能解释断点。实现必须落在本项目可审计代码与本地数据模型内。

## 目标

本次要完成：

- 将盘古记忆划分为三层：
  - 历史会话采集层：主路径，面向 Codex 本地 SQLite / 会话文件中的真实用户消息，用于后续补采、去重和长期候选生成。
  - 注入实时监听层：辅助路径，负责当前页面最近用户输入、注入心跳、workspace/thread 识别、快速候选和前端状态同步。
  - 采集日志与自检层：裁判路径，记录历史读取、注入捕获、候选判断、DB 写入和状态同步的结果与原因。
- 增加会话采集日志层：只要注入脚本捕获到用户消息，就写入本地采集日志。
- 即使没有生成候选，也要记录捕获时间、workspace、消息长度、脱敏摘要、是否触发候选、未触发原因。
- 增强自动候选诊断，明确区分：
  - 没捕获消息
  - 文本不可学习
  - 重复候选
  - 候选写入失败
  - 数据库失败
- 增加盘古自检链路，分层检查：
  - 注入心跳
  - 最近用户消息捕获
  - 候选请求
  - DB 计数
  - 前端 runtime
  - manager 状态同步
- 增加注入摘要缓存：把已确认记忆、待确认候选、最近采集摘要整理成可注入上下文，并落盘为可审查的“会话启动必读摘要”。
- 增加可审查学习提取算法：基于明确规则/偏好/项目约定/工作流约束识别高置信记忆。
- 精进长期记忆提炼算法：识别“经验教训 / 踩坑复盘 / 失败后的下次规则 / 用户纠错后的稳定结论”，并以 `lesson-learned` 分类进入长期记忆。
- 增强相似记忆合并：学习写入前使用关键词重叠与稳定 SimHash 海明距离判断相似规则，优先更新旧条目而不是重复新增长期记忆。
- 高置信学习提取结果自动进入长期记忆；低置信或不可学习内容只写采集日志，必要时保留待确认候选能力。
- 复制出来的一次性终端命令输出、Git/PowerShell 错误日志、权限失败堆栈等临时诊断内容不得自动进入长期记忆；除非用户明确改写为项目规则或偏好，否则只保留采集证据。
- 管理工具必须提供显式“提炼长期记忆”入口，触发同一套历史采集、学习提取、脱敏过滤、相似合并和摘要缓存刷新链路。

本次不包含：

- 不把低置信、寒暄、系统上下文、截图附件说明自动写入长期记忆。
- 不记录 API key、Bearer token、`sk-` 等敏感原文。
- 不重置或删除现有 `memory_assist.sqlite`。
- 不修改 Claude 中文注入脚本。
- 不做无关 UI 重构。

## 用户视角描述

用户在 Codex 当前对话中输入消息后，即使盘古记忆判断这条消息不适合作为候选，也可以在管理工具或自检结果中看到：消息已被哪一层捕获、脱敏摘要是什么、为什么没有生成候选、最近一次候选请求和数据库计数是否正常。

当用户输入可学习的项目规则、偏好、修复要求或记忆自检指令时，盘古记忆应使用可审查学习提取算法判断置信度。高置信内容直接写入长期记忆；低置信内容只保留采集证据或待确认候选。

## 功能要求

- 注入实时监听层应在捕获到最新用户消息时调用后端采集接口，写入采集日志。
- 采集日志必须包含：
  - `workspace`
  - `source`
  - `source_session_id`
  - `captured_at`
  - `text_length`
  - `text_hash`
  - `summary`
  - `candidate_triggered`
  - `candidate_reason`
  - `skip_reason`
- 采集日志不得保存敏感原文；摘要必须脱敏并限制长度。
- 同一 workspace、同一消息 hash 的重复采集应去重或更新，不应无限刷库。
- 学习未触发时必须给出 `skip_reason`，例如 `not_learnable`、`duplicate_recent_memory`、`no_latest_user_text`。
- 学习提取算法必须在高置信规则判断前排除临时命令输出和错误日志，避免把 `git push` 403、`git credential-manager` 用法错误、PowerShell 提示符等一次性诊断文本误判为 `ui-rule` 或 `workflow-rule`。
- 学习提取算法必须覆盖经验教训信号，例如“以后/下次/不要再/不能再/仍然/还是/没变化/失败/原因/根因/教训/经验/踩坑/复盘”等表达。
- 经验教训类内容必须同时具备上下文对象和可执行动作信号，例如盘古记忆、manager、Codex、按钮、前端、构建、验证、日志、反馈、数据库、自检等对象，以及必须/需要/应该/先/再/不要/不能/修复/验证/构建等动作。
- 经验教训类长期记忆写入时应归类为 `lesson-learned`，并生成带“经验教训：”前缀的稳定文本，方便会话启动摘要直接注入。
- 自动长期写入成功或失败后应更新采集日志，记录学习结果。
- 自检应返回分层状态，而不是只返回 0 条：
  - `injection`
  - `capture`
  - `candidate`
  - `database`
  - `runtime`
  - `manager`
- session 摘要应包含已确认记忆、待确认数量和最近采集摘要，供注入层展示和后续上下文注入使用。
- 后端应在本地应用状态目录维护一个可审查的盘古注入摘要缓存文件，内容只能来自 `memory_assist.sqlite` 和真实 Codex 会话采集结果。
- Codex 注入脚本启动时即使当前页面没有可查询文本，也应调用 session 摘要接口，确保“会话启动必读摘要”被刷新。

## 数据与接口要求

- 允许在 `memory_assist.sqlite` 中新增兼容迁移表，用于采集日志。
- 旧数据库必须可自动迁移，不得清空现有 `memory_items` 或 `memory_candidates`。
- 注入脚本新增或复用本地 bridge 接口：
  - 写入采集日志
  - 更新候选判断结果
  - 读取 session 摘要
- 后端接口返回值应使用真实 DB 计数，不使用前端本地估算。
- manager 状态中的“工作区”不得只统计长期记忆和待确认候选，还必须纳入：
  - `memory_captures` 中已经采集过的 workspace；
  - Codex 本地 SQLite / 会话目录中存在的真实会话 workspace；
  - 每个 workspace 的长期记忆数、待确认数、采集记录数、会话数和最近采集时间。
- 状态刷新可以执行有限数量的历史补采，但只能写入采集日志，不得在普通状态刷新时自动生成候选或自动批准长期记忆；会话启动摘要刷新、自检修复或显式自动学习路径可以把高置信学习提取结果自动写入长期记忆。
- “提炼长期记忆”属于显式自动学习路径，允许把高置信历史消息写入长期记忆；仍必须遵守不可学习过滤、敏感信息脱敏和相似记忆合并。
- 注入实时监听层识别 workspace 时，应优先使用真实项目路径或项目 id；当页面 DOM 暂时拿不到项目上下文时，应复用最近一次可信项目上下文，避免立即回落到 `codex:path:*`。
- `codex:path:*` 只能作为最后兜底标识，用于解释“当前页面无法稳定识别项目路径/thread/projectId”，不得被当成真实项目规范名称。
