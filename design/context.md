# Context

goal: 解决 Codex 模型菜单覆盖 CCP 注入弹窗的问题，并把容易与 CodexPlusPlus 等同类工具混淆的漫画面板重设计为有 CCP 产品识别度的本地控制舱。

user: 高频使用 Codex、第三方 API、插件和盘古记忆的本地开发者；希望一眼确认增强链路状态，同时不想进入复杂管理工具完成每次轻量切换。

JTBD: 在 Codex 内快速确认本地增强运行状态，调整模型、插件、会话和服务模式能力，并能明确理解这些能力由 CCP 本地控制、可审查且可回退。

constraints:
- stack: 单文件原生 JavaScript/CSS 注入，不引入依赖或网络字体。
- preserve: 所有现有 tab、开关、按钮、bridge route 和关闭隔离行为。
- scope: 只改 Codex 注入弹窗、契约测试和相关文档，不改管理工具主界面。
- brand: 突出本地控制室、模型桥接、盘古记忆、可审查/可回退，不模仿同类产品。
- host: 运行在 Codex 页面中，必须隔离样式并应对原生 portal 浮层。

success:
- 模型菜单不再显示在 CCP 弹窗上方。
- 2 秒内可识别 CCP、本机运行和三项核心能力。
- 长列表具备明确分区，关闭、导航和状态始终可见。
- 现有功能和事件代理无回归。
- 1280×800 与窄屏窗口均可使用。

scope v1:
- “盘古本地控制舱”视觉系统。
- 双栏场景导航、能力概览和功能分区。
- portal 层级修复、响应式和基础键盘焦点。

non-goals:
- 不重做 Codex 原生模型选择器。
- 不把完整管理工具搬进弹窗。
- 不新增盘古记忆数据展示或后端状态字段。

assumptions:
- 截图中的模型列表属于 Codex 临时 portal；最高层遮罩配合 Escape 收拢可以稳定消除覆盖。
- 用户更需要“本地运维控制台”的可信感，而不是娱乐化漫画表达。

risks:
- 深色控制舱不能降低长文本可读性，因此内容卡使用偏亮深灰并保持高对比。
- 首页能力标识不能伪装实时检测结果，因此只表达既有能力，不显示虚构数值。
