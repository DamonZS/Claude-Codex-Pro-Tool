# Context

goal: 为 Claude Codex Pro 管理工具探索一版“未来空间概念 UI”评审原型，先让用户逐页评审视觉方向，再决定是否应用到真实前端。

user: 管理工具的高频使用者与维护者，关注 Codex / Claude 启动状态、供应商、插件、会话、维护、设置和版本信息；熟悉当前功能入口，对误删或恢复旧功能非常敏感。

JTBD: 在不改变现有功能信息架构的前提下，快速判断每个管理模块的状态，并通过更强的空间感、层次感和未来交互美学评估新版视觉是否值得落地。

constraints:
- stack: 真实应用是 Rust + Tauri + React/Vite；本阶段只做独立静态预览。
- pages: 预览必须覆盖当前 7 个页面：概览、供应商、工具与插件、会话管理、维护、设置、关于。
- non-destructive: 不改真实 `App.tsx` 功能树、不改 Tauri command、不改后端、不改注入脚本。
- visual direction: 先构建智能管理工具的空间场景，再嵌入 UI；非平面应用；3D 立体、半透明磨砂玻璃悬浮面板、局部高饱和撞色、柔和渐变光影、强空间景深、高级赛博轻奢质感。
- density: 内容布局必须合理、不拥挤；不能把功能入口藏起来。

success:
- 每个现有页面都有可评审预览。
- 预览中的导航和页面名称与当前管理工具页面保持一致。
- 视觉明显区别于普通扁平后台，呈现空间场景和悬浮面板。
- 预览不新增真实功能、不删除现有功能，也不暗示不可实现能力。
- 用户确认后，才能进入真实前端应用阶段。

scope v1:
- 独立 HTML/CSS/JS 评审原型。
- 页面导航、主要状态卡、模块卡、表单/列表/开关/按钮等视觉样例。
- 桌面优先，并提供基础响应式。

non-goals:
- 本阶段不把设计应用到 `apps/claude-codex-pro-manager/src/App.tsx`。
- 不接入真实后端数据。
- 不启动或停止 Codex / Claude。
- 不修改安装包或发布配置。

open assumptions:
- 预览内容使用静态模拟数据，仅用于视觉和布局评审。
- 用户优先评审整体美学、页面密度、空间层次和功能入口保留情况。

risks:
- 未来空间感如果过强，可能压过运维工具的可读性；预览需控制面板密度和文字对比。
- 若用户认可视觉方向，落地到真实前端时仍需逐页映射现有组件，避免再次引入旧功能或删错入口。

iteration notes:
- V1 被用户否定，主要风险判断为：空间海报感过强、背景压过信息、页面像概念展示而不是可用管理工具。
- V2 方向改为“可用界面优先的未来空间工作台”：完整文字侧栏、状态区 + 主工作区 + 辅助面板结构、减少纯装饰光效、保留 3D 透视网格和悬浮玻璃质感。
- V2 明确不恢复用户之前要求删除的设置卡片，不把真实前端功能增删带入评审原型。
