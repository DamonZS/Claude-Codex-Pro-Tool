# Codex 注入弹窗设计方向

## A — 盘古本地控制舱（采用）

principle: 把 CCP 表达为 Codex 的本地 AI 运维层，而不是另一个插件设置页。

layout: 顶部品牌与连接状态；左侧场景导航；右侧能力概览和分区控制；深石墨骨架配青绿能量轨、琥珀诊断色。

optimizes: CCP 品牌识别、状态可信感、长列表扫描和桌面高频操作。

sacrifices: 相比浅色表单更有视觉存在感，需要控制光效避免压过正文。

fit: 最强。直接映射 README 中“本地控制室”、第三方模型增强、盘古记忆、可审查和可回退原则。

## B — 原生 Codex 检查器

principle: 尽可能融入 Codex 原生组件，像系统 Inspector 一样安静。

layout: 浅色单栏、分组列表、原生 segmented control。

optimizes: 学习成本和宿主一致性。

sacrifices: 品牌辨识最弱，仍容易与其他增强工具雷同。

fit: 中等。可用但不能解决用户提出的“突出个性化和特点”。

## C — 盘古命令中枢

principle: 搜索/命令优先，用动作而非页面组织能力。

layout: 顶部命令框、最近动作、状态时间线，设置进入二级抽屉。

optimizes: 熟练用户速度和扩展性。

sacrifices: 改变现有信息架构和交互习惯，实施风险高，轻量弹窗会变成复杂应用。

fit: 适合作为后续版本，不适合本次“保留功能、最小业务改动”的边界。

## 决策

采用 A。它在不改后端和设置模型的情况下，最大化 CCP 差异化；B 无法满足品牌目标，C 超出本次范围。
