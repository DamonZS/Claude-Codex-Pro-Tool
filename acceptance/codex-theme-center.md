# 验收标准：Codex 主题中心

验证对象：`spec/codex-theme-center.md`

## 验收前提

- 使用包含本功能最终代码的工作区构建，不以静态原型、Mock 数据或浏览器预览替代真实 Tauri 应用。
- 涉及本机 Codex 文件的现场验证必须先记录受影响文件及其哈希并建立可恢复备份；证据不得包含 API Key、Token 或其他完整凭据。
- 自动化写入测试优先使用临时 Codex 目录和故障注入夹具，不覆盖用户真实配置。

## 通过标准

### 1. 独立一级导航与页面入口

- 左侧一级导航存在独立、常驻且可识别的“主题中心”入口，不隐藏在设置、维护、工具或二级菜单中。
- 点击入口进入真实主题中心路由，刷新页面、应用启动后的路由恢复和外部导航均不会回落到其他页面；选中态与页面标题正确。
- 主题中心页面有清晰可见的“新建主题”入口。点击后打开真实的主题导入流程，取消操作不会产生主题记录、临时文件或成功提示。
- 证据：前端路由/导航契约测试、源码检查、原生 Tauri 交互录像或截图。

### 2. 默认列表、排序与图片卡片

- 首次进入主题中心默认展示“全部主题”，不要求用户先选择分类、清除筛选或输入搜索内容。
- 内置主题与已成功导入的主题均出现在默认列表；空搜索、重新进入页面和重启管理工具后结果一致。
- 常用桌面宽度下主题列表每行恰好三张图片卡片；至少在 `1440x900` 与 `1920x1080` 原生窗口中验证三列。
- 较窄窗口可按断点降为两列或一列；在 `960x640` 下不得出现卡片互相覆盖、图片拉伸、文字遮挡、不可达操作或页面横向溢出。
- 有合格预览的卡片仅展示由真实 Codex 主 Renderer 采集的 PNG；预览记录中的主题 ID / 版本、Codex 版本 / 构建指纹、运行时载荷 SHA-256 和截图 SHA-256 均与当前卡片及本机 Codex 一致，且卡片实际读取图片的 SHA-256 与验收截图相同。
- 主题版本、Codex 构建、运行时载荷或截图文件任一变化后，旧预览立即失效并显示稳定的“等待真实运行截图”或等价空态；不得回退到设计稿、静态示意图、其他主题图片或旧版本截图。
- 图片 / 空态、选中态、长主题名和状态文案不会改变网格列宽或造成布局跳动；卡片预览不得暴露绝对路径或使用 `file://`。
- 证据：组件/样式检查、预览版本失效测试、截图哈希 / 采集记录核对，以及上述尺寸的原生应用截图和对应真实 Codex 截图。

### 3. Codex 默认主题永久置顶

- 第一张卡片始终是“Codex 默认主题”，并有明确的默认主题标识；此顺序不受主题导入、名称排序、筛选清除、应用切换、页面刷新或管理工具重启影响。
- 默认主题是内置保留项，不从用户导入目录伪造，不允许删除、覆盖、重命名或被同 ID/同名导入项替换。
- 当前未应用自定义主题时，默认主题卡片显示为当前生效；应用自定义主题成功后，只有对应主题显示为当前生效。
- 主题库仅有默认主题时，页面同时保留首位默认主题卡片，并在其后显示带“新建主题”操作的明确空态；不得显示空白网格、伪造主题或用空态替换默认卡片。
- 当前 Codex 构建没有默认主题真实截图，或已有截图属于旧构建 / 已损坏时，默认卡片显示“暂无同版本真实截图”或等价预览空态，同时仍可表达默认主题恢复状态。
- 证据：排序/保留 ID 单元测试、重复或冲突导入测试、仅默认主题组件测试、预览缺失 / 过期测试和原生应用交互记录。

### 4. 点击主题会真实应用

- 点击任一自定义主题卡片会调用后端应用命令，并真实更新 Codex 使用的主题资源或主题注入配置；只改变前端选中态或本地 UI 偏好不算通过。
- 后端成功返回后，重新读取主题状态能得到同一活动主题；关闭并重开管理工具后活动状态仍正确。
- 应用成功时界面明确提示“需要重启 Codex”或同等含义的文案；若提供重启按钮，它必须复用现有真实 Codex 重启动作。
- 使用新构建的管理工具应用主题并重启 Codex 后，Codex 界面呈现所选主题；至少以可识别区域的前后截图和受控主题产物哈希变化证明，不以 Toast 单独作为证据。
- 重复应用当前主题具备确定结果，不会叠加重复注入片段、重复资源或持续增长文件。
- 含 `css_variables`、`root_attributes` 与 `asset_variables` 的主题应用后，真实 Renderer 中的样式节点、变量、根类 / 根属性、Data URI、主题 ID 和 generation 与规范化运行时载荷逐项一致，不多不少；重新读取的载荷哈希与后端编译哈希一致。
- 证据：后端定向测试、应用前后文件/状态差异、Loader 运行时快照、管理工具提示截图，以及同一真实 Codex 验收会话重启前后的 PNG 截图。

### 5. 默认主题真实清理并恢复

- 从自定义主题切换到“Codex 默认主题”会调用后端恢复命令，删除或停用所有由主题功能写入的活动覆盖、生成资源和主题专属注入内容，而不只是把前端活动 ID 改为默认值。
- 恢复后重新查询状态、重启管理工具并重启 Codex，均保持默认主题；Codex 界面与应用自定义主题前的官方默认外观一致。
- 恢复操作可重复执行并保持幂等，不因目标文件已清理而报伪失败，也不删除已导入的主题库内容或其他功能拥有的文件。
- 恢复只删除 Loader 实际新增且当前仍为其写入值的类、属性、变量、Data URI、主题节点和 CCP 标记；对 Loader 曾覆写且未发生外部漂移的值恢复应用前值，对预先存在、由其他能力拥有或应用后被外部改动的状态保持不变并记录所有权冲突。
- 证据：应用再恢复的集成测试、主题管理产物清单/哈希、所有权状态矩阵，以及同一真实 Codex 验收会话恢复前后的 PNG 截图。

### 6. 失败路径不得误报成功

- 列表、导入、应用、恢复任一后端操作失败时，命令返回失败状态和可诊断但不泄露敏感信息的错误；前端显示失败反馈，不显示“应用成功”“恢复成功”或要求重启以生效的成功提示。
- 应用失败时活动主题、卡片选中态和磁盘上的上一可用主题保持不变；刷新页面后不得显示失败目标为当前生效。
- 恢复失败时不得把默认主题标记为当前生效；仍须如实展示原活动主题或明确的未知/需修复状态。
- 至少覆盖不存在的主题 ID、损坏资源、无效导入包、目标不可写、写入中断、提交后状态复核失败、Data URI 编译失败、Loader 自修复后复核失败和所有权冲突。
- 证据：前后端失败契约测试、故障注入测试和失败态截图。

### 7. 主题导入校验与事务写入

- 导入在任何持久化写入前完成整体校验，至少覆盖：清单严格反序列化、必填字段存在、受控 `css_variables` / `root_attributes` / `asset_variables` schema、主题 ID / SemVer / 展示名称合法、真实截图元数据、资源显式声明、路径规范化、类型 / 扩展名 / 魔数 / MIME 一致、资源可解码，以及文件数量和单文件 / 分类 / 解包 / 生成载荷大小不超限。
- 主题 ID 必须满足规格正则，CSS 变量、根类、根数据属性和资产变量必须满足各自 CCP 命名空间、数量、值长度和保留键规则；拒绝未知字段、错误 JSON 类型、重复规范化键、分号逃逸、声明块、`url()` 混入 `css_variables`、任意根属性和未声明资产。
- 路径段只接受 `^[A-Za-z0-9][A-Za-z0-9._-]{0,79}$`；拒绝保留 / 重复 ID、空段、`.` / `..`、反斜杠、绝对 / UNC / 盘符 / ADS 路径、目录逃逸、符号链接 / 重解析点 / 硬链接、通配符、NUL / 控制字符、尾随点或空格、Windows 保留设备名、Unicode NFC 或大小写规范化冲突、超过 8 层 / 240 UTF-8 字节的路径和非法文件名。错误指出失败类别且不回显不必要的绝对路径。
- 只接受规格列出的 JSON、CSS、PNG、JPEG、WebP、WOFF2；扩展名、解析结果、魔数和 MIME 任一不一致即失败。SVG、HTML、JavaScript、可执行文件、嵌套归档、未知类型、远程 URL、`file:`、作者提供的 `data:`、`blob:`、脚本协议和 `@import` 均被拒绝。
- 对 8 层 / 240 字节路径、80 字节路径段、32 MiB 压缩包、64 MiB 解包总量、256 个文件、256 KiB manifest、2 MiB 单 CSS / 4 MiB 总 CSS、8 MiB 单图 / 40 百万解码像素、5 MiB 单 WOFF2、32 MiB 原始图片总量和 48 MiB Data URI 运行时载荷逐项测试“恰好上限通过、超过一层 / 一个字节 / 一个文件 / 一个像素失败”；限制由后端按实际读取结果计算。
- CSS 本地图片引用和 `asset_variables` 在暂存阶段全部转换为 MIME 正确的规范 Data URI；转换后重新解析的运行时载荷不存在本地路径、`file://`、远程 URL、未声明资源或未验证 Data URI，且相同输入在重复运行和支持平台上产生相同 SHA-256。
- 每次导入、应用、更新和恢复都有持久化 `Mutation Journal`（变更日志）或等价可恢复记录。记录至少包含操作 ID、操作类型、目标主题、阶段、前后版本/哈希、暂存及备份的受控相对位置、开始/更新时间和最终状态，不包含凭据、CDP 连接密钥或文件正文。
- 日志在首次破坏性动作前原子持久化，并按“已校验 -> 已建立快照 -> 已写入暂存 -> 已原子替换 -> 已复核 -> 已提交”或语义等价的阶段推进；日志更新本身崩溃时仍能判断最后一个完整阶段。
- 导入和应用必须遵循“临时目录完整写入 -> 从临时目录重新读取并验证 -> 同文件系统原子替换 -> 提交后复核”的顺序，不得逐文件直接覆盖正式主题目录。
- 原子替换前保留上一稳定版本及其完整性信息；新版本通过提交后复核前不得删除上一版本。完成后至少保留一个可用于回滚的上一版本，清理策略不得先删后换。
- 替换或更新已有用户主题时先建立可恢复备份。提交任一步失败会依据日志自动回滚到导入前状态，旧主题和当前活动主题仍可使用；暂存文件与不完整目录被清理。
- 覆盖更新当前正在生效的主题后，活动主题 ID 保持不变、`generation` 恰好递增一次，重新读取的运行时载荷来自新 manifest、CSS 与资源；覆盖未生效主题时 `generation` 保持不变。该行为由核心回归测试覆盖。
- 对 Windows 文件占用/共享冲突、权限拒绝、杀毒软件短暂占用和原子重命名失败有明确处理：可进行有上限的重试；最终失败时返回实际失败，不更新当前主题，不删除上一版本，不误报成功。
- 若文件占用同时阻止回滚，日志必须保留“提交失败且待恢复”状态及恢复所需信息；释放占用后再次执行恢复或重启管理工具，系统能完成回滚并将状态收敛到上一稳定版本。
- 进程在每个事务阶段中断后，下次启动都能根据日志恢复旧版本或完成已验证提交，并清理无主暂存内容；半成品不得进入主题列表或被标为当前生效。
- 成功导入后主题只出现一次，并保持默认主题第一张；仅当预览采集记录通过主题版本、Codex 构建、运行时载荷及截图哈希复核时显示截图，否则显示待采集空态。成功结果来自提交后的重新读取或等价状态复核。
- 测试至少覆盖：验证前失败、暂存写入失败、原子替换失败、提交后复核失败、替换目标被独占打开、回滚目标被独占打开、进程在各日志阶段中断，以及释放占用后的自动/手动恢复。
- 证据：每类合法 / 非法 manifest 夹具、路径 / 命名 / 类型 / MIME / 魔数 / 精确大小边界表、Data URI 确定性测试、临时目录集成测试、恶意 / 损坏夹具测试、各阶段故障注入、Mutation Journal 脱敏样例、导入前后目录树 / 哈希和回滚恢复结果。

### 8. 与现有 Codex 能力隔离

- 应用、切换、导入和恢复主题均不改变汉化开关 / 资源、模型目录与模型选择注入、输入框增强、菜单增强、供应商选择、协议映射、Base URL、API 配置或凭据。
- 若主题与其他 Codex 增强共用注入文件，主题写入只能修改明确归属主题的受控区域；不得覆盖、截断、重复或重排翻译、模型、供应商和 API 配置相关内容。
- 主题运行时样式使用唯一、稳定且明确归属主题功能的 `<style>` 节点或等价独立挂载点。它不得复用汉化、模型选择、输入框、菜单、状态条或其他注入功能的 style/script 节点、MutationObserver、事件监听器或 DOM 扫描入口；应用主题只更新主题自有状态，恢复默认只撤销 Loader 拥有的状态。
- 重复应用、Renderer 重连和 Codex 多次重启后，目标 Renderer 中最多存在一个主题 style 节点；汉化、模型、输入框与菜单注入节点的 ID、属性、内容哈希、顺序、监听器数量和可用状态保持不变。
- DOM 隔离测试需预置汉化、模型、输入框、菜单和状态注入节点，再执行主题应用、更新、损坏修复和恢复；除 Loader 专属且有所有权记录的主题状态外，DOM 快照和内容哈希不得变化。
- 在“主题应用 -> 切换供应商/模型 -> 恢复默认主题”和“先配置供应商/模型 -> 应用主题 -> 重启 Codex”两条顺序下，各功能状态和真实行为均保持正确。
- 自动化测试对主题操作前后的非主题配置做结构化比较或哈希比较；敏感字段仅比较是否保持一致，不输出原值。
- 主题命令不得读取、写入、复制、备份或更新 Provider 凭据文件与凭据字段。对 `auth.json`、供应商配置、Base URL、API Key、Header、Bearer Token 等使用哨兵值和文件访问审计验证：内容、哈希、大小、修改时间及访问记录均无主题操作造成的变化。
- 日志、错误、IPC 返回、Mutation Journal、截图、测试快照和主题 manifest 中不得出现 Provider 凭据原文、完整认证 Header、敏感 Base URL 查询参数或可复用 Token。
- 现有 Codex 注入、供应商和模型相关契约测试继续通过。
- 证据：DOM / 监听器隔离测试、输入框输入与菜单交互回归、文件访问审计、凭据文件前后元数据 / 哈希、受影响文件 diff、既有回归测试命令输出和最小现场操作记录。

### 9. CDP 动态端口与 Codex Renderer 身份校验

- 运行时应用或复核主题时通过现有启动器/helper 状态、进程信息或 CDP 发现机制取得当前动态端口，不硬编码 `9222` 或沿用未复核的历史端口。
- 连接 CDP 前校验监听端口归属当前 Codex 进程；至少关联规范化可执行文件/应用包身份、PID、进程启动时间和预期父子关系，排除普通 Chrome、Edge、Claude Desktop、旧 Codex 进程及其他 Electron 应用。
- 从 CDP target 列表选择 Renderer 时校验 target 类型、URL/页面来源和 Codex 运行时指纹。指纹至少组合两个彼此独立的稳定信号，例如 Codex 专属 DOM 根、受控全局标记、应用版本/资源指纹或已知 bundle 特征；仅依赖窗口标题不算通过。
- 在实际插入/替换主题 style 节点前再次校验端口、PID、进程启动时间、target ID 和 Renderer 指纹。端口复用、进程重启、target 销毁、指纹变化或候选目标歧义时立即停止，不向未确认 Renderer 注入，不更新活动主题，不提示成功。
- 多窗口或多 Renderer 场景采用明确且可测试的目标策略：只处理全部通过身份与指纹校验的 Codex 主 Renderer，并记录脱敏 target 标识；不得把 DevTools、后台页、登录 WebView 或其他应用页面当作主题目标。
- Renderer 重建后可通过动态发现重新挂载唯一主题 style 节点；重新发现不得导致主题、汉化或模型节点重复。
- 测试至少覆盖：随机动态端口、陈旧端口、端口被其他进程复用、同机多个 Electron 应用、Codex 进程重启、Renderer target 更换、伪造标题但指纹不匹配，以及多个合法 Codex 窗口。
- 证据：进程/CDP 夹具测试、目标选择测试、脱敏运行日志，以及来自通过校验的真实 Codex Renderer 的截图。

### 10. 前后端命令契约

- 前端主题列表、导入、应用和恢复默认主题的每个 `invoke` 名称都与 Tauri 注册命令完全一致；不存在只定义未注册、只注册未调用或浏览器端伪实现。
- 参数字段名、可空性和序列化格式与 Rust 命令入参一致；主题 ID 与导入来源均由后端再次校验，前端校验不作为信任边界。
- 返回结构遵循项目统一命令结果约定，明确区分成功、失败、消息与数据；列表项至少能稳定表达主题 ID、名称、预览资源、是否内置及是否当前生效。
- 应用和恢复命令仅在持久化提交及状态复核成功后返回成功；前端只依据真实命令结果更新活动状态和 Toast。
- 命令错误可序列化到前端，未知字段、缺失参数、未知主题和后端 I/O 错误均有契约测试；错误不导致前端未处理异常或永久加载态。
- 证据：Rust 命令测试、Manager `windows_subsystem` 契约测试、TypeScript 类型检查和源码注册表检查。

### 11. 加载、空态、可访问性与交互稳定性

- 加载中、仅默认主题、预览待采集 / 已过期、导入中、应用中、恢复中和错误状态均有明确反馈；进行中的主题操作会阻止重复提交且不会让整页布局跳动。
- “仅默认主题”状态保留默认卡片并在其后显示导入空态；“默认主题缺少同版本截图”只替换默认卡片的预览区域，不隐藏卡片、不伪造截图，也不与“默认主题不可恢复”状态混淆。
- 图片加载失败、哈希不匹配或版本不匹配有可识别占位和错误语义，不把损坏主题显示为可成功预览，也不回退到旧图或 `file://`。
- 卡片可通过键盘聚焦和触发，当前生效状态不只依赖颜色；图片有替代文本，图标按钮有可访问名称或 Tooltip，焦点环可见。

- 使用上述默认 `target\release` 产物启动管理工具；视觉证据来自原生 Tauri 窗口，不以静态 HTML、设计稿或仅 Vite 浏览器页面替代。

- 应用前、自定义主题生效后、恢复默认后三张 Codex 截图必须来自同一验收会话；自定义主题卡片实际显示的预览必须与该版本生效截图为同一哈希对象，默认卡片实际显示的预览必须与恢复默认后的截图为同一哈希对象。
- 人工检查所有截图中的文字、图片、卡片、按钮和 Toast，无重叠、截断、拉伸、非预期滚动条或低对比不可读内容。

### 14. 非默认主题强制注入与重启证据

- 在关闭所有其他 Codex 前端增强项、当前主题为默认主题时，Launcher 生命周期保持不启动 helper、不执行注入的既有行为。
- 在相同设置下切换为任一非默认主题后，Launcher 生命周期必须启动 helper、执行注入并启动 bridge watchdog；不得只更新主题状态文件。
- Launcher 已运行时再次触发启动，以及 Manager 点击“启动 / 重启 Codex”，均使用当前主题状态参与注入判定。
- Windows 请求 CDP 端口可绑定时保持该端口；端口被正常进程或系统僵尸监听占用时选择非零动态端口。测试必须证明实际端口进入 Launcher 状态并可被后续注入链使用；动态分配失败不得把 `0` 作为 CDP 端口。
- Manager 重启监控接受本次重启时间边界之后由 Launcher 报告的实际动态端口，并拒绝时间边界之前的旧状态；实际端口与请求端口不同不得导致 `launch_ports` 阶段超时。
- 主题新文档脚本进入 Renderer 后，目标页面中出现且仅出现一个 `claude-codex-pro-codex-theme` 样式节点，节点记录的主题 ID 与 generation 和本地活动主题状态一致。
- 含 `root_attributes` 的非默认主题进入 Renderer 后，`document.documentElement` 精确包含 manifest 声明的 CCP 根类 / 根数据属性以及专属主题 ID / generation 标记；Loader 不通过扫描 CSS 文本猜测根类。样式节点内容相同但任一受控根状态缺失或错误时，再次注入会主动修复而不是报告 `unchanged`。
- 真实 Renderer 验证必须读取至少一个受主题控制区域的 `getComputedStyle`，证明背景、前景色或其他明确主题属性已变化；仅存在样式节点或注入返回 `ok` 不算通过。
- 恢复默认主题后，专属主题样式节点、CSS / 资产变量、Data URI、CCP 专属主题数据标记和 Loader 实际拥有的根属性 / 根类全部清理或按所有权记录恢复；Loader 未拥有或已被外部改变的 Codex 根状态，以及汉化、模型、输入框、菜单和其他增强节点保持不变。
- 诊断日志能区分主题判定、CDP 端口上线、Renderer 注入确认与超时失败，且不包含主题 CSS 全文、凭据或无关私密数据。
- 证据：Launcher 端口选择与生命周期回归测试、Manager 本次启动状态边界测试、主题注入源码契约测试、Manager 重启诊断契约测试、脱敏运行日志、Loader 运行时快照和真实 Codex Renderer 检查。

### 15. Theme Loader 精确应用、损坏自修复与所有权

- 使用同时声明 `entry_style`、`css_variables`、`root_attributes` 和 `asset_variables` 的完整夹具生成规范运行时载荷；Loader 首次应用后逐键、逐值、逐哈希比对完全一致，未声明状态不被新增，上一主题中由 Loader 拥有但新主题未声明的状态被准确移除。
- `asset_variables` 和 CSS 本地图片引用在真实 Renderer 中均为 MIME 正确的 Data URI；运行时 DOM、CSSOM、网络请求和控制台记录中不存在主题库存绝对路径、`file://`、远程回退或读取本地图片的请求。
- 分别破坏专属 style 节点（删除、复制、改文）、主题 ID / generation、每类 CSS 变量、每类根类 / 根属性和每类资产 Data URI 后触发重连 / 新文档 / 周期复核；Loader 对每种情况返回 `repaired`，恢复精确载荷并通过二次哈希 / 计算样式复核。
- 修复后再次复核返回稳定的 `healthy` / `unchanged`，不重复写 DOM、不新增 style 节点、MutationObserver 或事件监听器；修复失败返回失败，不更新活动主题或预览截图。
- 所有权矩阵至少覆盖：应用前不存在、应用前同值已存在、应用前异值被覆写、应用后仍为 Loader 写入值、应用后被 Codex 或其他增强改值。切换 / 恢复只撤销 Loader 拥有且未漂移的状态，恢复可恢复原值，并对外部改值保持原样、报告 `ownership_conflict`。
- 恢复前后对 `documentElement`、汉化、模型、输入框、菜单、状态条及其他增强节点做结构化快照和监听器计数；除 Loader 自有状态外无变化，输入框内容 / 焦点和菜单开闭 / 点击行为仍正确。
- 证据：规范化 manifest 与运行时载荷快照、Data URI MIME / 哈希清单、逐项损坏注入测试、二次复核结果、所有权矩阵、DOM / CSSOM / 监听器差异和真实 Codex 前后截图。

### 16. 真实主题视觉一致性与运行时资源闭环

- 在通过动态端口、进程身份和 Renderer 指纹校验的真实 Codex 主 Renderer 中检查当前主题。页面中必须出现且仅出现一个 CCP 主题 style 节点；节点主题 ID、generation 和运行时载荷哈希与本地当前主题状态完全一致。
- 对 manifest 声明的每个 `css_variables`、`asset_variables` 以及 CSS 本地图片逐项取值。CSS 变量不得为空，资产变量必须是 MIME 正确的 Data URI 或等价受控内存资源；至少在一个真实目标元素的 `getComputedStyle` 中被实际引用，相关图片在 Renderer 内解码成功且 `naturalWidth` / `naturalHeight` 或等价固有尺寸大于零。仅变量存在但最终 `background-image` 为 `none`、资源未被引用或解码失败判定为失败。
- 逐项核对主题视觉契约：品牌层存在性与位置、背景图片哈希及其定位 / 缩放、关键主辅色的计算值、主要区域顺序与尺寸关系、主题根类 / 根属性。品牌、背景、色彩或布局中任一项只出现在卡片图片而未进入真实 Codex，均判定为静态假预览并失败。
- 卡片原始预览应与当前版本真实 Codex 截图为同一 SHA-256。若卡片使用裁切、缩放或脱敏派生图，必须能追溯到该真实截图并提供确定性派生参数；若采用“与实装逐项一致”的表示，必须提交相同 Codex 构建、主题版本、载荷和基准视口下的逐项一致性矩阵及真实 Codex 对照截图，独立设计稿、静态 HTML / CSS 或手工拼图不算通过。
- Loader 仅在唯一 style、主题 ID / generation、全部 CSS / 资源变量、图片解码、关键计算样式和视觉契约全部通过时返回 `active: true` 或等价当前生效状态。任一项失败必须返回 `failed`、列出脱敏差异类别，并保持 `active: false`；Manager 不显示当前生效，不登记或更新预览。
- 模拟资产变量为空、Data URI 截断、图片解码失败、style 重复、品牌层缺失、背景计算值为 `none`、关键色值错误和布局标记缺失。每一种情况都必须明确失败；发生在主题切换 / 更新提交后时依据 Mutation Journal 回滚到上一稳定主题，并重新证明上一主题唯一 style、资源变量和 `active` 状态健康。
- 证据：真实 Renderer 的 CDP 检查结果、style 节点计数、去除 Data URI 正文后的 CSS / 资源变量清单、图片 MIME / SHA-256 / 解码尺寸、`getComputedStyle` 摘要、`active` 状态、视觉契约逐项对照、真实 Codex PNG、预览派生记录及失败回滚日志。

## 失败条件

出现以下任一情况即视为未通过：

- 主题中心不是独立一级导航，缺少新建主题入口，或首次进入没有展示全部主题。
- 默认主题不是永久第一张，或可被导入主题覆盖、删除、重命名、排序到后面。
- 常用桌面窗口不是三列图片卡片，或任一验收尺寸出现遮挡、溢出、图片拉伸和不可操作内容。
- 主题卡片使用设计稿、Mock、旧主题 / 旧 Codex 版本、载荷哈希不一致或无法追溯到真实 Codex Renderer 的图片；预览缺失 / 过期时没有进入明确空态。
- 仅有默认主题时默认卡片被空态替代、没有导入空态，或默认主题无同版本截图时借用其他图片。
- 点击主题只改变前端状态，未真实写入 Codex 主题；成功后未提示重启 Codex。
- 非默认主题未独立触发完整注入链，或 Manager 重启后没有重新向 Renderer 注册当前主题脚本。
- 恢复默认主题仅修改活动 ID，遗留主题覆盖或主题注入仍在 Codex 中生效。
- 后端失败后仍显示成功、错误主题被标为当前生效，或破坏上一可用状态。
- manifest 未受控支持 `css_variables`、`root_attributes`、`asset_variables`，或导入未严格校验路径 / 命名 / 类型 / 魔数 / MIME / 数量 / 大小、缺少 Mutation Journal 或等价可恢复记录、未经暂存复核便覆盖正式目录、未保留上一版本、留下半成品，或占用 / 中断失败后旧主题不可恢复。
- 任一本地图片以路径、`file://` 或远程 URL 进入卡片 / Renderer，Data URI 未经类型和大小校验、非确定性，或转换失败后仍提交主题。
- CDP 使用硬编码或陈旧端口，未校验 Codex 进程与 Renderer 指纹，或向身份不明的 Electron Renderer 注入主题。
- 主题复用或改写汉化、模型、输入框或菜单注入节点 / 扫描器 / 监听器，重复产生主题 style 节点，或恢复默认后仍残留 Loader 自有主题状态。
- Loader 没有精确应用 manifest、未修复自身缺失 / 重复 / 错值状态、修复失败仍报成功，或切换 / 恢复时清理了不属于自身的类、属性、变量、节点或外部改值。
- 主题图片只存在于主题包、状态文件或预览图中，没有在真实 Codex Renderer 的计算样式中被引用并成功解码；品牌层、背景、色彩或布局只做了静态预览；必需资源失败后仍返回 `active`、更新预览或未回滚上一稳定主题。
- 主题操作读取、改变或泄露翻译、模型、供应商、API 或 Provider 凭据配置。
- 前后端命令名称、参数或返回结构不一致，或 Tauri 命令未注册到最终应用。
- 未通过工作区测试和默认 `target\release` 全量构建，或缺少原生视觉、同版本真实 Codex PNG 与 Codex 重启闭环证据。
