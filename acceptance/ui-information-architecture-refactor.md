# 验收标准：CCP 管理工具视觉与信息架构重构

验证对象：`spec/ui-information-architecture-refactor.md`

## 通过标准

### 1. 可回滚基线

- `codex/backup-ui-refactor-baseline-20260714` 存在并指向实施前提交 `838cf480126e25df62b975f7f709d754ab7a953a`。
- 重构未覆盖或回滚基线中的 Codex 注入、重启、凭据同步等业务修复。
- 证据：`git show-ref`、`git diff` 和业务契约测试。

### 2. 新信息架构与旧路由兼容

- 左侧最多显示七个一级入口：概览、供应商与路由、客户端与增强、会话与记忆、插件/Skills/MCP、维护与诊断、设置；不得再显示“模型与协议”一级入口。
- `overview`、`supplier`、`tools`、`sessions`、`memory`、`maintenance`、`settings`、`about` 原路由仍可解析。
- `memory` 与 `sessions` 共用“会话与记忆”一级选中态，并可在域内切换。
- `about` 与 `settings` 共用“设置”一级选中态，并可在域内切换。
- 外部初始化路由和 `claude-codex-pro-navigate` 事件继续有效。
- 证据：源码检查、前端类型检查和 Windows 契约测试。

### 3. 应用外壳和命令栏

- 存在左侧导航、52–56px 顶部命令栏、主工作区和可选详情布局。
- 展开侧栏为 232–248px，折叠为 64px，且折叠状态不导致导航不可识别。
- 顶部显示面包屑/页面、Agent 筛选、当前供应商、代理健康和全局动作。
- 顶部最右侧同时可见“启动/重启 Codex”“启动/重启 Claude”“Claude 一键汉化”三个图标加文字命令，点击分别复用原有真实 action；其他控件排列在它们左侧。
- `Ctrl+K` / `Cmd+K` 可打开命令搜索，键盘可选择并跳转页面，Escape 可关闭。
- 保留 `ops-shell`、`ops-rail`、`ops-workspace`、`ops-topbar`、`ops-screen` 类名。
- 证据：源码检查、原生应用交互和截图。

### 4. 主题与视觉 Token

- 提供浅色、深色、跟随系统三种主题，并在本地持久化选择。
- 浅色和深色使用规格规定的背景、表面、边框、文字、系统蓝和状态色。
- Windows 首选 Segoe UI，macOS 首选系统字体；模型、路径、URL、日志使用等宽字体。
- 卡片/面板圆角不超过 8px，按钮/输入为 6px，字距为 0，字体不按视口缩放。
- 普通内容面板无装饰阴影；表格、表单、日志和长文本保持高不透明度。
- 最终界面不以紫蓝渐变、光球、网格、霓虹、营销 Hero 或卡片套卡片构图。
- 证据：CSS Token 源码检查和浅深色截图。

### 5. 模型映射与协议编辑

- 不存在重复的“模型与协议”一级导航；旧 `models` 导航值安全回落到供应商页。
- Codex 与 Claude 供应商继续在原供应商编辑器中编辑协议、显示名称、实际请求模型和上下文。
- 两类供应商的“实际请求模型”选择器行为和视觉一致；当前值可读，菜单候选项可点击和键盘选择，长模型 ID 不遮挡相邻行。
- 浅色选择器和 Portal 菜单使用浅色高不透明表面，深色主题使用深色表面；不可用和选中状态同时提供文字或描边。
- 证据：源码检查、浅深色原生应用交互与截图。

### 6. 客户端与增强页面

- 页面复用现有 Codex、Claude Desktop、Claude 中文补丁、Watcher 和设置状态。
- 现有启动/重启、汉化、连接修复和维护动作保持调用原 actions。
- “已安装”“已启用”“当前生效”“健康状态”具有不同且包含文字的表达。
- Agent 筛选会真实过滤客户端内容。
- 证据：源码检查、契约测试和原生应用截图。

### 7. 现有业务页面完整性

- 供应商 CRUD、排序、切换、模型发现、连接测试、凭据诊断、Claude Desktop 预览/写入仍存在。
- 会话列表、上下文查看、删除、导出/修复和 Provider Sync 行为仍存在。
- 盘古记忆的项目接续、新项目指南、搜索、编辑、归档、恢复、确认、导入导出与高级诊断仍存在。
- 插件、Skills、MCP 的筛选、状态、风险、安装、卸载、修复和开关入口仍存在。
- 维护、设置、日志、检查更新、下载并运行安装包和关于信息仍存在。
- 会话页首行仅展示整行“历史会话修复”，下方 Codex 与 Claude 会话管理为 1:1 双栏；不再显示重复的“会话管理”介绍卡。
- Codex 与 Claude 插件仓库在常用桌面宽度为 1:1 双栏；工具、插件和 Skills 列表的正文、状态徽标与 Agent 图标达到清晰可辨的对比度。
- 证据：`windows_subsystem` 测试、源码检查与代表页面截图。

### 8. 状态、可访问性和响应式

- 加载、空、错误、禁用、离线、部分成功、无权限和脏数据状态保留已有文字反馈。
- 状态不只依赖颜色；图标按钮有可访问名称或 Tooltip；焦点环清晰。
- 支持 `prefers-reduced-motion`，非必要动画被关闭。
- 在 960×640、1280×800、1440×900、1920×1080 下，无关键文字溢出、控件遮挡、布局跳动或不可达滚动区。
- 窄窗口下侧栏可折叠，复杂内容允许纵向滚动或受控横向滚动。
- 证据：原生应用多尺寸截图和人工检查记录。

### 9. 隐私与产品边界

- 截图、日志和新增 UI 不显示完整 API Key、Token、敏感地址、私密会话全文或不必要的绝对路径。
- 产品文案不表达官方账号接管、托管、模拟、订阅或支付管理。
- “恢复官方配置”仅表示本地配置状态。
- 可见产品品牌使用 `CCP`；“盘古记忆”仅作为记忆功能名称出现。
- 证据：源码与截图检查。

### 10. 构建与回归

以下命令全部成功：

```powershell
npm --prefix apps/claude-codex-pro-manager run check
npm --prefix apps/claude-codex-pro-manager run vite:build
cargo fmt --check
cargo test -p claude-codex-pro-manager --manifest-path Cargo.toml --test windows_subsystem -- --nocapture
cargo test --workspace
cargo build --release
```

- 若 Manager、Launcher、Codex 或构建进程占用目标文件，先核实进程可执行文件绝对路径，只结束本项目对应进程后重跑。
- 最终 Release 产物必须写入仓库默认 `target\\release`，不能只生成到自定义 `target-rebuild`。
- 证据：命令输出、产物路径与修改时间。

### 11. 原生视觉验收

- 使用原生 Tauri 窗口或系统默认外部浏览器，不使用 Codex 内置浏览器。
- 至少提供以下最终截图：首页、供应商、供应商编辑器、会话/记忆、扩展中心、设置页。
- 至少覆盖浅色与深色；至少覆盖 960×640 和一个常用桌面尺寸。
- 截图显示真实应用数据或真实空态，不使用静态概念稿替代。
- 证据：仓库外或验收目录中的 PNG 文件及人工检查记录。

## 失败条件

出现以下任一情况视为未通过：

- 删除或绕过现有 Tauri 命令、业务状态或功能入口。
- 旧路由或外部导航事件失效。
- 新页面使用模拟供应商、模型、客户端健康数据冒充真实结果。
- 界面在最小窗口发生关键控件重叠、不可操作或文本遮挡。
- 浅色主题仍出现硬编码深色输入/下拉，或模型候选项无法辨认和选择。
- 只完成静态稿、只构建 Vite，或未生成默认 `target\\release` 应用。
- 截图泄露完整凭据、敏感地址、私密会话全文或无关本地路径。

## 非目标

- 不验证第三方服务可用性或模型实际授权。
- 不新增或重构代理协议转换后端。
- 不迁移用户配置、会话或记忆数据库。
- 不改变安装注册、Release 发布或远端更新服务协议。
